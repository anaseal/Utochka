# Документация проекта: Graphic Editor for Silyanka Design

Проект представляет собой специализированный параметрический редактор для создания схем традиционных бисерных украшений (силянок). Система имитирует физику и геометрию сетчатого плетения, позволяя дизайнеру работать с логической структурой изделия, а не с простыми пикселями.

## 1. Основные сущности системы

### **Bead (Бисерина)**
Фундаментальный объект данных. Система разделяет их на две функциональные группы:
*   **NODE (Узел):** Ключевые точки пересечения нитей. Именно они определяют каркас изделия.
*   **SPAN (Соединительная):** Бисерины, заполняющие пространство между узлами на нити.

### **Grid Structure (Сетка)**
Математическая модель, построенная на чередовании рядов узлов.
*   **Четный ряд:** Базовая горизонтальная линия узловых точек.
*   **Нечетный ряд:** Ряд, имеющий горизонтальное смещение на 50% шага относительно четного ряда, что создает ромбовидную ячейку.

### **Design Map (Карта дизайна)**
Реестр состояний (`Record<string, string>`, ID бисерины → цвет), хранящий информацию о цвете каждой закрашенной бисерины. Карта полностью отделена от геометрии: изменение `width`, `height`, span-параметров пересчитывает координаты, но не затрагивает раскраску — цвет сохраняется до тех пор, пока существует бисерина с тем же ID.

## 2. Геометрия и логика плетения

### **Система рядов и колонн**
В отличие от таблиц, здесь расчет ведется по узловым точкам (NODE):
*   **Колонна (Column):** Вертикальная ось узлов. Параметр `width` (внутреннее состояние, `gridSize.width`) определяет количество узлов в чётном ряду; в нечётном ряду (со сдвигом на полшага) их на один меньше — это и есть число «колонок», которое реально считает линейка (`CanvasRulers`).
*   **Ряд (Row):** Горизонтальная ось узлов. Параметр `height` (внутреннее состояние, `gridSize.height`) определяет количество «этажей» из ромбов. Один полный ромб по вертикали формируется за счёт прохода через три уровня узлов (Верх–Центр–Низ); полное число рядов узлов равно `2 * height + 1`, из них чётных (промаркированных линейкой) — `height + 1`.
*   **Ширина/высота в хедере vs. внутреннее состояние:** Степперы Width/Height в `Header` показывают и принимают числа, совпадающие с линейкой, а не сырые `gridSize.width/height` — иначе цифра в хедере не совпадала бы с тем, что физически видно на холсте (при `width=30` линейка насчитывает 29 колонок, при `height=8` — 9 рядов). Пересчёт `±1` сделан в одной точке — [src/App.tsx](App.tsx) (`silyankaProps.gridWidth/gridHeight/onSetWidth/onSetHeight`); `generator.ts`, `mirror.ts` и остальная геометрия по-прежнему работают с исходными `gridSize.width/height`. Кнопки ± (`onWidthChange`/`onHeightChange`) меняют `gridSize` напрямую дельтой — сдвиг на константу не требует пересчёта.
*   **Самый верхний ряд (Top Edge):** Особая технологическая цепочка бисерин, соединяющая узлы самого первого ряда (r=0) дугой (не строго по горизонтали — см. «Дуга кромок» в разделе 3). Она служит «кромкой» изделия.
*   **Самый нижний ряд (Bottom Edge):** По умолчанию — просто последний ряд узлов. Независимый от Top/Bottom Edge и per-row override'ов декор `BottomEdgeDecor { enabled, span }` может добавить аналогичную верхней горизонтальную цепочку ниже последнего узлового ряда — см. «Нижняя цепочка (Bottom Chain)» в разделе 3.

### **Параметры ромба (Span Control)**
Система позволяет создавать асимметричные ячейки. Span-параметры можно менять как глобально, так и для конкретного ряда:
*   **Top Span (плечи):** Количество бисерин в верхних гранях ромба. Диапазон 3–10, регулируется кнопками ± «Top Edge» в хедере. При нажатии кнопки Reset (иконка ↻) глобальное значение топ-спана вернётся к дефолту (`beadsInSpan = 3`), а все per-row override для нечётных рядов (плечи, `r % 2 !== 0`) удаляются.
*   **Bottom Span (ножки):** Количество бисерин в нижних гранях ромба. Диапазон 3–10, регулируется кнопками ± «Bottom Edge» в хедере. Reset кнопка вернёт дефолт и очистит per-row override для чётных рядов (ножки, `r % 2 === 0`).
*   **Per-row Override:** Для каждого span-ряда (промежуток между двумя соседними рядами узлов) на левом краю канваса отрисовываются собственные ± кнопки и счётчик. Они переопределяют глобальные Top/Bottom Edge только для этого ряда; диапазон тот же — 3–10. Цветовая кодировка:
    *   **Нижние пролёты** (r чётный, ножки) — амбер.
    *   **Верхние пролёты** (r нечётный, плечи) — синий.
    *   **Верхняя горизонтальная грань** (r=−1) — также синяя (плечи), имеет собственный override.
    *   Если ряд переопределён локально, его счётчик выделяется более ярким цветом, чтобы отличить от наследуемого значения.
    Переопределения хранятся отдельно в реестре `rowSpanOverrides` ([src/App.tsx](App.tsx): `usePersistedState('silyanka:rowSpanOverrides', ...)`) и в общем случае **не сбрасываются** при последующем изменении глобальных Top/Bottom Edge. Исключение: если изменение глобального параметра делает override равным новому дефолту ряда, такой избыточный override удаляется (`pruneRedundantOverrides` в `updateTopSpan`/`updateBottomSpan`) — иначе он «протух» бы и навсегда заморозил ряд, поскольку `resolveSpanCount` отдаёт приоритет override'у. Reset же кнопки специально отсеивают override'ы для соответствующих рядов по чётности.
    Логика преобразования span-параметров в зависимости от чётности ряда централизована в [src/utils/spans.ts](utils/spans.ts): функция `resolveSpanCount(r, topSpan, bottomSpan, rowSpanOverrides)` — возвращает эффективное количество бисерин для пролёта ряда с учётом override'а.
*   **Compression:** Вертикальное сжатие (`verticalCompression = 0.2`) имитирует реальную форму бисера, который под собственным весом стремится принять форму приплюснутого овала, а не идеального круга.

### **Плотность сетки (Spacing)**
Базовое расстояние между узлами `spacing` (px) регулируется в хедере степпером «Spacing» (рядом с Zoom, есть также в overflow-панели для узких экранов). Диапазон **45–85px**, шаг **5px** (`BEAD_THEME.constraints.minSpacing/maxSpacing/spacingStep`, [src/config/theme.ts](config/theme.ts)), дефолт — 65px. Значение хранится в `GridConfig.spacing` ([src/App.tsx](App.tsx): `usePersistedState('silyanka:gridSize', ...)`) и напрямую передаётся в `generateSilyankaGrid` — генератор уже был параметричен по `spacing`, изменение плотности не требует правок геометрии, только масштабирует `stepX`/`getYStep`/`decorRowStep` (см. ниже). ViewBox канваса считается динамически от bounding box бисерин, поэтому корректно подстраивается под любую плотность.

### **Адаптивный шаг сетки**
Геометрические шаги рассчитываются на лету и могут не совпадать с базовым `spacing`:
*   **Горизонтальный шаг (`stepX`):** Берётся как максимум из `spacing * horizontalStepMultiplier`, `(internalTop + 1) * minBeadPitch` и `(internalBottom + 1) * minBeadPitch`, где `internalTop = topEdgeCount - 2` (верхняя кромка, r=-1), `internalBottom = bottomEdgeSpan - 2` (нижняя цепочка, 0 если она выключена), а `minBeadPitch ≈ 14px` — минимальный шаг без перекрытия (`spanRadius * 2 + 2`). Это автоматически раздвигает колонны, если верхняя или нижняя кромка содержит слишком много бисерин, чтобы поместиться в базовое расстояние.
*   **Вертикальный шаг (`getYStep`):** Считается индивидуально для каждого ряда `r` как `(getInternalCount(r) + 1) * spacing * verticalCompression`. Поскольку `getInternalCount(r)` зависит от фактического span на ряду (с учётом per-row override), высота строки динамическая.

## 3. Функциональные возможности

### **Динамический генератор (`generator.ts`)**
При любом изменении параметров (ширина, высота, глобальные span, per-row overrides) система мгновенно пересчитывает координаты каждой из сотен бисерин и заново выстраивает связи между узлами, формируя целостное полотно. Алгоритм работает в два прохода: сначала формируется сетка узлов (`nodeGrid`), затем по ней интерполируются SPAN-бисерины — вдоль верхней кромки (r=0), вдоль диагональных граней ромбов и, при включённом `BottomEdgeDecor.enabled`, вдоль нижней кромки.

#### **Дуга кромок (Top/Bottom Edge Arc)**
Верхняя кромка (r=0) и нижняя цепочка (при включении) — не плоская «полка» с постоянным Y-офсетом, а дуга: каждая внутренняя бисерина смещается от прямой между узлами по `sin`-профилю. Для бисерины `i` (позиция `1..count`), `t = i / (count + 1)`:

`y += direction · edgeArcHeight · sin(π · t)`

где `direction = -1` для верхней кромки (дуга наружу, вверх от полотна) и `direction = +1` для нижней. `edgeArcHeight = 13` ([src/config/theme.ts](config/theme.ts)). У узлов (`t → 0` и `t → 1`) смещение стремится к нулю, у середины пролёта (`t = 0.5`) — максимально; это не даёт бисеринам наложиться друг на друга при большом числе бисерин в кромке (в отличие от прежнего константного офсета). Реализовано функцией `generateArcSpan` в [src/utils/generator.ts](utils/generator.ts), отдельно от `generateSpan` — последняя по-прежнему используется для диагональных граней ромбов, без дуги (`yOffset = 0`).

#### **Нижняя цепочка (Bottom Chain)**
Аналог верхней горизонтальной кромки, добавляемый **ниже** последнего узлового ряда (`r = 2·height`), той же дугой. В отличие от Top/Bottom Edge и per-row override'ов, полностью независима от них: хранится в отдельном персистентном стейте `BottomEdgeDecor { enabled: boolean; span: number }` ([src/types/bead.ts](types/bead.ts); `usePersistedState('silyanka:bottomEdgeDecor', ...)` в [src/App.tsx](App.tsx)) — не читает и не пишет `rowSpanOverrides`/`topSpan`/`bottomSpan`. `span` (диапазон 3–10, дефолт 3) задаёт число бисерин напрямую: `internalBottom = max(0, span - 2)`. ID бисерин — `span-edge-bottom-link-${c}-bead-${i}` (не содержат номер ряда, поэтому стабильны при изменении `height`). Поддержка:
*   **Flood Fill:** нижняя цепочка подключена к графу смежности (`floodFill.ts`) через конечные узлы `node-{lastR}-c` и `node-{lastR}-{c+1}`, как и верхняя кромка.
*   **Mirror Mode:** зеркалирование по той же логике, что и у верхней кромки (`mirror.ts`), с `internalBottom = bottomEdgeDecor.span - 2`.
*   **UI:** тумблер «Bottom Chain» — в боковой панели `PendantsSidebar` (первый раздел панели, над Pendants и Decor), включает/выключает `enabled`. Степпер количества бисерин (±, диапазон 3–10) — на канвасе в `CanvasRulers`, в той же позиции под последним рядом узлов; управляет `bottomEdgeDecor.span` напрямую, минуя `rowSpanOverrides` и общий Bottom Edge из хедера.
*   **Взаимоисключение с подвесками:** Bottom Chain и точечные подвески (`pendantPlacements`) не могут быть включены одновременно — тумблер задизейблен, пока на канвасе есть хотя бы одна подвеска (`toggleBottomEdgeEnabled` в [src/App.tsx](App.tsx)); каталог шаблонов подвесок, в свою очередь, задизейблен, пока включён Bottom Chain (`PendantsSidebar.tsx`).

### **Промежуточный декор (Decor Bands)**
Помимо подвесок (концевой декор), силянка поддерживает **промежуточный декор** — горизонтальные полосы бисерин, вставляемые *между рядами узлов*.
*   **Модель:** Реестр `decorBands` ([src/hooks/useSilyankaProject.ts](hooks/useSilyankaProject.ts): `usePersistedState('silyanka:decorBands', ...)`; `Record<number, number>`, ключ — узловой ряд `r`, значение — число декор-рядов полосы `N`). Полоса с ключом `r` располагается в промежутке между узловым рядом `r` и `r+1`. Хранится отдельно, переживает перезагрузку (`localStorage`, ключ `silyanka:decorBands`). Минимальное значение для активной полосы — 1; значение 0 или отсутствие ключа означают отсутствие полосы (диапазон хранения: 1–10 строк).
*   **Геометрия:** Полоса состоит из `N` горизонтальных рядов бисерин, повторяющих разметку (ширину и смещение) узлового ряда `r`. Шаг между декор-рядами равен внутреннему шагу пролётов: `spacing · verticalCompression`. Полоса **раздвигает полотно по вертикали**: узловой ряд `r+1` и всё ниже опускаются на `N · шаг`; число колонн не меняется. Нижний декор-ряд полосы становится «виртуальным рядом `r`» — диагональные грани ромбов к ряду `r+1` строятся от него (координаты граней рассчитываются генератором с учётом расширения). Декор-бисерины имеют тип `SPAN` и ID вида `decor-{r}-{k}-{c}` (где `k` — индекс декор-ряда в полосе, от 0 до N−1), поэтому красятся карандашом/ластиком и учитываются в `CanvasStats` как обычные пролёты.
*   **Управление:** В сайдбаре (раздел «Decor», последний из трёх разделов панели, без переключения вкладок) находится карточка «Band». Её перетаскивают на зазор между рядами на канвасе — во время drag `CanvasRulers` подсвечивает ближайший зазор (розовая полоса); при сбросе полоса добавляется (или удаляется, если уже существует — toggle). Диапазон `N` — 1–10 (`decorDefaults.minRows = 1`, `decorDefaults.maxRows = 10`); после размещения счётчик полосы регулируется кнопками ± в списке «Placed» в том же разделе. Верхняя горизонтальная грань (r=−1) декор не поддерживает. Для вычисления зазоров сайдбар получает проп `rowGaps: {row,midY}[]`, вычисляемый в `App.tsx` из NODE-бусин.
*   **Mirror Mode:** Покраска декор-бисерины зеркалит парную по логическому зеркалу колонн (как у NODE): `col c ↔ col (width-1-c)` для чётных рядов, `col c ↔ col (width-2-c)` для нечётных.
*   **Изменение сетки:** Ключи `decorBands` привязаны к рядам узлов, а не к колоннам, — смена `width` их не затрагивает. При уменьшении `height` полосы на исчезнувших рядах удаляются ([src/hooks/useSilyankaProject.ts](hooks/useSilyankaProject.ts): общий хелпер `applyHeight` — вызывается из `updateDimension('height', ...)` и `setHeightAbsolute` — отфильтровывает `decorBands` так, чтобы остались только ряды `r < 2·newHeight`).

### **Инструменты редактирования**

#### *Карандаш (Pencil)*
Базовый режим. Закрашивание происходит «потоком»: достаточно зажать кнопку мыши и провести курсором по бисеринам — все, через которые проходит указатель, получают активный цвет. Запись в Design Map: `designMap[id] = activeColor`. Явно активируется кнопкой `Pencil` в хедере (первая в панели инструментов, перед ластиком) — клик по ней всегда возвращает в режим карандаша; кнопка подсвечивается как активная, когда `activeTool === 'pencil'`.

#### *Ластик (Eraser)*
Активируется кнопкой ⌫ в хедере. В этом режиме клик по бисерине удаляет её из Design Map (`delete designMap[id]`), возвращая в незакрашенное состояние по умолчанию. Поскольку ластик не использует цвет, выбор любого цвета (палитра, custom или recent) выводит из режима ластика обратно в карандаш; если же был активен инструмент, использующий цвет (заливка, штамп), выбор цвета лишь меняет активный цвет и не переключает инструмент.

#### *Заливка (Flood Fill)*
Активируется кнопкой `PaintBucket` в хедере (между ластиком и Mirror Mode). Повторный клик возвращает инструмент в режим карандаша. При активном режиме курсор канваса меняется на `crosshair`. Один клик по бисерине закрашивает всю связную область бисерин **того же эффективного цвета** активным цветом за один шаг истории (Undo отменяет всю заливку разом).

**Алгоритм:** BFS по графу смежности бисерин, построенному из массива `beads`. Связи:
*   **SPAN-цепочки** (верхняя кромка и диагональные грани ромбов): каждый кластер образует линейную цепь `startNode → bead-1 → … → bead-N → endNode`, где конечные узлы определяются из cluster ID (например, `edge-top-link-{c}` → `node-0-{c}` и `node-0-{c+1}`).
*   **Декор-полосы**: вертикальная смежность между рядами `k` и `k+1` одной колонки + горизонтальная смежность между соседними колонками в пределах одного ряда `k`. Декор-полосы **изолированы** от основной сетки (нет рёбер между `decor-*` и `node-*`/`span-edge-*`), поэтому заливка не «перетекает» из полосы в сетку и наоборот.

**Цвет старта:** `designMap[id] ?? defaultColorFor(bead.type)`. Если эффективный цвет стартовой бисерины уже равен активному цвету — заливка не выполняется (нет снимка в историю).

**Mirror Mode:** если режим включён, заливка выполняется также для зеркальной бисерины; оба результата объединяются в один вызов `remapDesignMap`.

#### *Штамп (Stamp)*
Активируется кнопкой `Stamp` в хедере (сразу после Flood Fill). Позволяет выделить рамкой раскрашенный мотив и многократно переносить его в другое место сетки, пока инструмент не переключат.

**Взаимодействие** (реализовано в `CanvasView`, вся геометрия — в [src/utils/stamp.ts](utils/stamp.ts)):
*   **Drag (сдвиг мыши > 4px, независимо от zoom)** — рисует рамку выделения и по отпусканию кнопки захватывает все бисерины (любого типа — узлы, пролёты, декор, кромки), чей центр (`bead.x/y`, без учёта радиуса) попал внутрь прямоугольника. Координаты рамки считаются через `getScreenCTM()` того `<g>`, что несёт `translate(offsetX, offsetY)`, — сразу в системе координат бисерин. В момент, когда драг распознан (превышен порог), протухший preview предыдущего узора сразу гасится, чтобы не мешать видеть новую рамку.
*   **Click (без сдвига) при загруженном узоре** — берёт ближайшую NODE-бисерину к точке клика как целевой якорь и ставит копию узора. Инструмент остаётся активным с тем же узором — повторные клики ставят новые копии, одна запись в историю (Undo) на одну постановку.
*   **Новый drag поверх уже загруженного узора** заменяет его новым выделением.
*   **Живое превью:** пока узор загружен, наведение на канвас подсвечивает бисерины целевой позиции (переиспользован проп `highlighted` у `BeadView`).
*   **Escape** сбрасывает загруженный узор (без выделения ничего не происходит). **Alt** делает то же самое — быстрый способ сразу вернуться в режим выделения новой рамки, не отпуская мышь ради Escape. **Переключение на любой другой инструмент** (карандаш/ластик/заливка/выбор другого цвета через `selectColor`) делает то же самое — обёртка `setSilyankaTool` в `App.tsx` сравнивает старый и новый `activeTool`. Так при каждом новом заходе в штамп нет «залипшего» узора от прошлой сессии, который иначе сразу показывал бы превью и мешал выделить новую область.
*   **Точка привязки (top/bottom):** по умолчанию `targetAnchor` совмещается с верхним рядом захваченного узора (`anchorRow`/`anchorCol`). Пока узор загружен, у кнопки `Stamp` в хедере появляется маленький кнопка-бейдж (`ArrowUpToLine`/`ArrowDownToLine`) на её углу — визуально дочерний тоггл, а не самостоятельный инструмент в общем ряду (`.tool-btn-group` / `.tool-btn-group__badge` в `Header.css`). Клик по бейджу **или** клавиша **Shift** (тап, не удержание — эквивалент клика, слушается в `App.tsx` вместе с Escape/Alt, `e.repeat` отсекает авто-повтор) переключают `stampAnchorEdge` насовсем — на нижний ряд узора (`anchorRowBottom`/`anchorColBottom`, тоже вычисляется в `captureStampPattern`) и обратно. Так можно навести курсор на существующий узел у верхнего края полотна и поставить туда именно *низ* штампа, а верхняя часть узора, «уехавшая» выше ряда 0, естественно обрежется (см. финальную защиту по `beadIds` ниже) — без необходимости заводить курсор за пределы канваса. Оба способа переключения полностью эквивалентны — состояние одно на весь сеанс штампа, а не временный оверрайд на одну постановку (это обсуждалось и было отклонено: удерживать модификатор ради каждого клика неудобно).
*   **Mirror Mode:** после постановки штампа каждая успешно перенесённая бисерина дополнительно прогоняется через `mirrorBeadId` — штамп ведёт себя симметрично карандашу.

**Модель трансляции id** (`translateBeadId(id, dRow, dCol, ctx, anchorRow)`, аналог `mirrorBeadId` для сдвига, а не отражения) — у сетки нет равномерной решётки, поэтому перенос id пересчитывается по правилам, а не сложением чисел:
*   `node-{r}-{c}` → `node-{r+dRow}-{c+dCol+correction}`. `correction` компенсирует то, что нечётные ряды физически сдвинуты на `stepX/2` относительно чётных (`generator.ts` — `isShifted`): при нечётном `dRow` (перенос через ряд другой чётности) бисерины, чья исходная чётность ряда `r` отличается от чётности ряда-якоря узора (`anchorRow`), получают `±1` к колонке — без этого узор рвётся на несвязанные «чётную»/«нечётную» половины, съезжающие друг относительно друга на целый шаг сетки. При чётном `dRow` `correction = 0` всегда. Та же коррекция применяется к диагональным граням (`span-edge-{r}-{c}-...`) и декору (`decor-{r}-{k}-{c}`) — они используют ту же систему координат `(r, c)`, что и NODE.
*   `span-edge-{r}-{c}-{side}-bead-{i}` (диагональные грани ромба) — origin-узел переносится как NODE (включая ту же коррекцию колонки); индекс бисерины внутри грани переиндексируется пропорционально: `t = i/(srcCount+1)`, `i' = round(t·(dstCount+1))`, где `srcCount`/`dstCount` — число бисерин в пролёте исходного/целевого ряда (та же формула, что `getInternalCount` в `generator.ts`). Это покрывает перенос между рядами с разным per-row override.
*   `span-edge-top-link-{c}-bead-{i}` и `span-edge-bottom-link-{c}-bead-{i}` (верхняя/нижняя кромка) — жёстко привязаны к одному ряду (r=0 и r=2·height), поэтому переносятся **только по горизонтали** (`dRow === 0`); при вертикальном сдвиге штампа эти бисерины выпадают из вставки, остальные переносятся как обычно. Это свойство модели, а не баг.
*   `decor-{r}-{k}-{c}` → `decor-{r+dRow}-{k}-{c+dCol+correction}` (та же коррекция колонки, что у NODE), допустимо только если у целевой полосы `decorBands[r+dRow] >= k`.
*   `pendant:*` не поддерживается (отдельная система `colorMap`, вне Design Map).
*   Финальная защита для всех правил: кандидат принимается, только если он реально присутствует в текущем массиве `beads` (`Set<string>` по id) — подстраховка от ошибок в ручной геометрии; частично «свисающая» за край сетки вставка просто теряет недостижимые бисерины, без визуальных артефактов.

`captureStampPattern(ids, beads, designMap)` фиксирует **два** anchor'а — минимальный и максимальный `(row, col)` среди захваченных NODE-бисерин (или среди всех, если узлов в выделении нет): `anchorRow/anchorCol` (верхний ряд выделения) и `anchorRowBottom/anchorColBottom` (нижний ряд) — и список `{id, color}` **только для реально закрашенных бисерин** (`designMap[id] !== undefined`). Незакрашенные бисерины внутри рамки выделения в узор не попадают — штамп переносит рисунок, а не «допечатывает» пустоту поверх того, что уже может быть на месте вставки (иначе `designMap[id]` получил бы буквальное значение цвета-плейсхолдера, что ломает проверку `isEmpty` в `BeadView` и рендерится некорректно). `applyStampPattern(pattern, targetAnchor, ctx, edge = 'top')` выбирает пару anchor по `edge` (`'top'` → `anchorRow/anchorCol`, `'bottom'` → `anchorRowBottom/anchorColBottom`), считает `dRow/dCol` относительно неё и прогоняет каждую запись через `translateBeadId`, отбрасывая недостижимые id; результат — один патч `{id: color}`, применяемый через `remapDesignMap` за один шаг истории.

#### *Custom Color Picker*
Кнопка с конической градиентной заливкой в палитре открывает скрытый системный `<input type="color">`. Выбранный цвет становится активным и автоматически включает карандаш. Если активный цвет не входит в фиксированную палитру, кнопка показывает этот цвет вместо градиента — это служит индикатором «работаем кастомным цветом».

#### *EyeDropper («Пипетка»)*
Доступна только в браузерах, поддерживающих нативный `EyeDropper API` (`'EyeDropper' in window`). Кнопка отрисовывается условно. Клик вызывает системный селектор пикселя — выбранный цвет (`sRGBHex`) подставляется как активный, инструмент переключается в карандаш. Отмена выбора обрабатывается беззвучно.

#### *Recent Colors*
Динамическая секция из 5 слотов в палитре — отдельная строка под базовыми цветами (двухэтажная раскладка `.palette__grid`: сверху 5 базовых цветов, снизу 5 recent-слотов, симметрично). Любой цвет, выбранный через Custom Color Picker, EyeDropper или клик по уже существующему recent-слоту, попадает в начало списка; дубликаты дедуплицируются (повторный выбор того же цвета не сдвигает порядок и не создаёт второй слот); при превышении лимита самый старый вытесняется (FIFO). Цвета из базовой палитры **не** записываются в recent — это исключает засорение списка. Список переживает перезагрузку страницы: сериализуется в `localStorage` под ключом `silyanka:recentColors`, при загрузке валидируется (массив hex-строк формата `#RRGGBB`, длина обрезается до лимита; невалидные данные молча игнорируются). Пустые слоты отрисовываются как круги с пунктирной рамкой и не реагируют на клики (`pointer-events: none`).

#### *Подвески (Pendants)*
Система для добавления декоративных подвесок на нижние узлы сетки:
*   **Шаблоны подвесок** ([src/data/pendantTemplates.ts](data/pendantTemplates.ts)): Предопределённый набор из 8 шаблонов (Drop, Rhombus, Cluster, Fringe, Net, Net with cluster, Drop with cluster, Bugle bead). Каждый шаблон состоит из массива бусин с указанием смещения (`dx`, `dy`) от якорной ноды, формы (`circle` или `rect`) и типа (`NODE` или `SPAN`).
*   **Размер подвески** ([src/data/pendantTemplates.ts](data/pendantTemplates.ts)): Шаблоны заданы в макетных координатах, где обычная бусина имеет радиус 18. При рендере всё умножается на `PENDANT_SCALE = BEAD_THEME.sizes.nodeRadius / 18`, чтобы нормальная бусина совпала с размером ноды сетки. Шаблоны поддерживают несколько размеров: `R = 18` (обычная), `R_BIG = 23.5` (крупная фокусная), `R_BERRY = 11.5` (мелкая грозди), `R_CLUSTER = 14` (бусина тройной грозди).
*   **Размещение подвесок:** Реестр `pendantPlacements` (`PendantPlacement[]`, список объектов `{ placementId, templateId, col, colorMap }`). Подвеска крепится к нижнему узлу `col`-й колонны последнего ряда (`r = 2·height`). На каждой колонне может быть только одна подвеска — новая заменяет старую. Цвета отдельных бусин в подвеске хранятся в `colorMap` (индекс в `template.beads` → цвет). Размещение переживает перезагрузку страницы (`localStorage`, ключ `silyanka:pendantPlacements`).
*   **Взаимоисключение с Bottom Chain:** см. «Взаимоисключение с подвесками» в разделе «Нижняя цепочка (Bottom Chain)» — подвески и Bottom Chain не могут быть включены одновременно.
*   **Управление подвесками:** Боковая панель (Sidebar) справа с кнопками для добавления шаблонов на конкретные колонны. Клик по шаблону в панели добавляет его на выбранную колонну (определяется через `hoveredCol` при наведении мыши). Удаление осуществляется нажатием на иконку в самом шаблоне на канвасе. Кнопка «×» справа от заголовка раздела «Pendants» очищает только подвески (`onClearAll`), не трогая Bottom Chain и декор-полосы; задизейблена при отсутствии размещений.
*   **Покраска бусин подвески:** При активном инструменте карандаш / ластик клик на бусину подвески покрашивает её активным цветом или стирает (как обычные SPAN-бусины). Обычное мышечное поведение — зажать кнопку и провести по нескольким бусинам для потокового рисования.
*   **Mirror Mode для подвесок:** В зеркальном режиме подвеска добавляется на обе симметричные колонны одновременно (логическое зеркало: `col c ↔ col (width-1-c)`). Покраска одной бусины также зеркалится. Удаление/перемещение одной подвески влияет на её парную (сохраняются как логическая пара для синхронизации).
*   **Трансформации при изменении сетки:** При уменьшении `width` подвески на исчезнувших колонках удаляются. В Mirror Mode при добавлении/удалении колонн все подвески сдвигаются вместе с рисунком (см. раздел 3 → *Зеркальный режим*).
*   **Боковая панель (PendantsSidebar):** Отрисовывается слева от хедера при нажатии кнопки с иконкой `Layers`. Содержит превью всех шаблонов, переключатель выбранной колонны (индикатор `hoveredCol`), кнопку удаления для каждого активного размещения и общую кнопку «Reset all». Каталог никогда не хранит цвет — превью бусин всегда рисуются в «пустом» стиле (см. ниже).
*   **Структура панели:** Над скроллящейся областью — закреплённый заголовок панели («Pendants & Decor»). Единая прокручиваемая панель без переключения вкладок, три визуально разделённых линиями-разделителями раздела сверху вниз — (1) тумблер Bottom Chain, (2) каталог подвесок (сетка 3 колонки, компактные карточки: превью ~52px, название до 2 строк), (3) декор-полосы (карточка Band + список «Placed» с ±). Каждый раздел имеет собственный заголовок и короткое англоязычное описание (`pendants-sidebar__section-title` / `-desc`): «Bottom Chain» — Decorative edge added below the last row; «Pendants» — Drag a template onto a bottom-row node; «Decor» — Drag a band onto a gap between rows. Все три раздела скроллятся как единая область (`pendants-sidebar__body`); футер с кнопкой «Reset all» и контекстной подсказкой закреплён снаружи скролла и не прокручивается вместе с содержимым.
*   **Точечная очистка разделов (`pendants-sidebar__section-clear`):** Справа от заголовка каждого из трёх разделов (`pendants-sidebar__section-heading-row`, `justify-content: space-between`) — компактная кнопка «×», очищающая **только** этот раздел, в отличие от общей «Reset all» в футере (которая сбрасывает все три состояния разом): «Bottom Chain» → вызывает `onBottomEdgeToggle` (задизейблена при `!bottomEdgeEnabled` — нечего выключать); «Pendants» → `onClearAll` (задизейблена при отсутствии размещений); «Decor» → `onClearDecor` (задизейблена при отсутствии активных полос). Подсказка про включение Bottom Chain при наличии подвесок теперь ссылается на точечную очистку раздела «Pendants», а не на общий «Reset all» (очистка одних только подвесок не обязана сбрасывать декор).
*   **Визуальный стиль бусин подвески:** Бусины подвески переиспользуют те же CSS-классы (`bead`, `bead--type-node`/`bead--type-span`, `bead--empty`, `bead__body`) и переменную `--bead-color`, что и обычная сетка (`BeadView`) — распределены по обёртке `<g>` (тип/пустота) и самому телу (`bead__body`), а не на одном элементе, иначе CSS-селекторы вида `.bead--empty .bead__body` (селектор потомка) не сработают. Благодаря этому непокрашенная подвеска выглядит как полупрозрачный контур (как пустой узел сетки) вместо невидимого `transparent`-круга, а покрашенный NODE получает то же свечение (`drop-shadow`). Каталог шаблонов в сайдбаре (`PendantPreview`, `BandPreview`) всегда рисуется в состоянии `bead--empty`, т.к. это шаблоны, а не размещения.


#### *Hitbox-адаптация*
Для удобства работы с мелкими элементами активная зона клика (`hitboxRadius = 11px`) значительно превышает визуальный размер бисерины (6.5px для NODE, 6.0px для SPAN).

#### *Зеркальный режим (Mirror Mode)*
Режим симметричного рисования, включается кнопкой в хедере (иконка `FlipHorizontal`). При активации на канвасе отрисовывается вертикальная пунктирная ось симметрии по центру (`x = maxX / 2`, где `maxX` — максимальная координата узлов чётного ряда). Любой клик или мазок по бисерине автоматически закрашивает и её зеркальную пару — оба изменения попадают в Design Map за **один** шаг истории (один Undo отменяет пару целиком). Работает и с ластиком: стирание зеркалит так же. Покраска бусин подвески тоже зеркалится.

Логика зеркала различается для двух типов бисерин ([src/utils/mirror.ts](utils/mirror.ts)):
*   **NODE — логическое зеркало:** 
    *   Чётные ряды (не смещены): `col c ↔ col (width-1-c)`.
    *   Нечётные ряды (смещены на `stepX/2`): `col c ↔ col (width-2-c)` (так как нечётный ряд содержит на одну колонку меньше). Крайний узел нечётного ряда (`col = width-1`) не имеет пары.
    Из-за того, что нечётные ряды смещены на `stepX/2`, зеркальные пары в нечётных рядах визуально не равноудалены от оси — это реальная геометрия плетения силянки.
*   **SPAN — геометрическое зеркало:** 
    *   **Верхняя кромка** (r=0): индексы бисерин внутри `internalTop + 1` переворачиваются симметрично: `bead_i ↔ bead_(internalTop+1-i)`.
    *   **Диагональные грани ромба** (рёбра между рядами): пролёты отражаются по фактической геометрии с обменом сторон `left ↔ right`.
    *   **Декоративные полосы** (`decor-r-k-c`): повторяют разметку узлового ряда `r`, поэтому логическое зеркало как у NODE.
    Логическое зеркало для пролётов невозможно — оно порождало бы грани, которых нет в графе силянки.

Состояние режима сохраняется в `localStorage` (ключ `silyanka:mirrorMode`). При `width ≤ 1` ось не отрисовывается и зеркалирование не выполняется.

**Изменение ширины в Mirror Mode:** при включённом режиме кнопки Width ± меняют ширину на **±2** (добавляют/убирают по колонке с каждой стороны), чтобы рисунок остался по центру относительно новой оси ([src/App.tsx](App.tsx): `updateDimension` → `shiftDesignMapColumns`). Design Map автоматически пересчитывается функцией [src/utils/regrid.ts](utils/regrid.ts): `shiftDesignMapColumns(designMap, delta, newW)` сдвигает все ID бисерин на соответствующее число колонок; бисерины, вышедшие за границу сетки, отбрасываются. Все подвески также сдвигаются вместе с рисунком (`col += delta`). Пересчёт записывается в историю и отменяется через Undo.


#### *Zoom System*
Масштабирование холста от 25% до 300% с шагом 10%. Реализовано CSS-трансформацией (`transform: scale(var(--canvas-zoom))`), поэтому SVG-границы остаются векторно-чёткими на любом уровне зума.

#### *Тема холста (Dark / Light)*
Переключатель тёмной/светлой темы **только для холста** — карточки `.canvas__svg` и её содержимого (бисер, линейки, ось зеркала). Хедер, сайдбар и внешняя рама (`.editor__viewport`) всегда тёмные: светлый холст читается как «лист бумаги» на тёмной раме.
*   **Состояние:** `canvasTheme: 'dark' | 'light'` в [src/App.tsx](App.tsx), персистентно в `localStorage` (ключ `silyanka:canvasTheme`). Дефолт при чистом хранилище — `dark` (системная тема не учитывается). Переключается плавающей кнопкой ☀/🌙 **у холста** — в правом нижнем углу над «Download PNG» (`.canvas-theme-toggle` в `CanvasView`), не в хедере.
*   **Механизм:** `data-canvas-theme={canvasTheme}` вешается в [CanvasView](components/Editor/CanvasView/CanvasView.tsx) **на два элемента**: на карточку `<div className="canvas__svg">` (отсюда токены каскадятся вниз ко всем бусинам/линейкам) и на раму `<main className="editor__viewport">` (ancestor карточки — CSS не умеет выбирать предка по потомку, поэтому фон рамы задаётся своим атрибутом). Тема реализована **через CSS-переменные**: тёмные значения — дефолт (в `.canvas__svg` и в `var(...)`-fallback у бисера/оси/чипов), светлые — переопределяются селектором `[data-canvas-theme="light"]`.
*   **Светлая палитра:** поверхность холста `#F5F5F7` (нейтральный светло-серый), граница/скроллбар — приглушённый slate. **Рама (`.editor__viewport`)** остаётся тёмной, но менее чёрной — navy `#0e1b33` вместо `#020617`, чтобы снизить контраст со светлой карточкой. Базовый (незакрашенный) бисер, чей видимый вид задаётся правилами `.bead--empty` в [BeadView.css](components/Editor/BeadView/BeadView.css) (в модели `defaultColorFor` = `transparent`), рисуется **чистым контуром без заливки и без свечения** (мутная полупрозрачная заливка и серый glow на светлом фоне выглядят «грязно»): span — мягкий slate-контур, нода — более тёмный и плотный. **Закрашенная нода** в тёмной теме подсвечивается glow собственным цветом (`drop-shadow(0 0 2px var(--bead-color))`) — «pop» ярких бусин; в светлой теме этот glow **приглушается** (радиус размытия 1px вместо 2px), иначе тёмные бусины на светлом фоне дают мутный ореол — «pop» ноды сохраняется, но мягче. Экспорт PNG повторяет это правило через поле `nodeGlow` в палитрах [exportScheme.ts](utils/exportScheme.ts). Ось зеркала (белый пунктир, невидимый на светлом) переопределяется в тёмный (`--canvas-mirror-axis`). Чипы span-контролов на белом осветляются (розовый/голубой tint вместо тяжёлых тёмных заливок).

### **История действий (Undo/Redo)**
Система записывает историю на уровне мазков, а не отдельных кликов. Стек живёт в [useDrawing.ts](hooks/useDrawing.ts) и покрывает **и** Design Map сетки, **и** размещения подвесок (`pendantPlacements`) — один снимок истории (`HistorySnapshot`) хранит связку `{ designMap, pendants }`, поэтому один Undo/Redo синхронно откатывает изменения обоих типов, сделанные за один жест (например, мазок карандашом, задевший и сетку, и подвеску, или заливка, окрасившая оба сразу):
*   **Запись в историю:** Состояние `{ designMap, pendants }` фиксируется в момент начала мазка (`startDrawing`, вызывается при `mousedown` на канвасе — до разделения на «клик по сетке» / «клик по подвеске»). При завершении мазка (`stopDrawing`, `mouseup`/`mouseleave`), если изменилась карта **или** размещения подвесок, сохранённый снимок добавляется в стек `past`. Операция `clearAll` тоже добавляет снимок в историю перед очисткой, но только если было что чистить (карта или хотя бы одна подвеска с непустым `colorMap`) — холостого снимка не возникает.
*   **Заливка (Flood Fill):** Использует общий метод `applyPatch(designMapFn, pendantsFn)`, который применяет обе функции-патчера (любая может быть `null`, если тип не задет) и пишет **один** снимок в историю, даже если заливка одновременно перекрасила бусины сетки и подвески (через якорную ноду).
*   **Стек истории:** Хранит до 30 снимков (`MAX_HISTORY = 30`). При переполнении самый старый снимок удаляется.
*   **Redo-стек:** Стек `future` сбрасывается при каждом новом мазке или очистке. Undo перемещает текущее состояние (обе части) в `future`, Redo — возвращает обратно.
*   **Управление:** Кнопки ↩ / ↪ в хедере (с блокировкой при пустых стеках). Клавиатурные сокращения работают как с `Ctrl`, так и с `Cmd` (macOS): `Ctrl/Cmd+Z` — отменить, `Ctrl/Cmd+Y` или `Ctrl/Cmd+Shift+Z` — повторить.
*   **Вне истории:** Структурные изменения — добавление/удаление подвески (`addPlacement`/`removePlacement`/`clearAllPlacements` в `usePendants`), изменение Width/Height/Top/Bottom Edge, декор-полос — в стек не попадают и Undo не отменяются (кроме сдвига `designMap` при Width ±2 в Mirror Mode, который идёт через `remapDesignMap` и пишется в историю; сопутствующий сдвиг колонок подвесок при этом в историю не попадает).

### **Интерфейс управления (Header)**
Фиксированная панель инструментов в верхней части экрана ([src/components/Editor/Header/Header.tsx](components/Editor/Header/Header.tsx)):
*   **Палитра цветов:** 5 хардкод-цветов (`#ff4757`, `#ffd32a`, `#22d3ee`, `#e879f9`, `#ffffff`). Клик по любому цвету активирует карандаш и устанавливает этот цвет как активный. Клики по базовой палитре **не** записываются в Recent Colors.
*   **Recent Colors:** 5 слотов второй строкой под базовой палитрой. Накапливают цвета, выбранные через Custom Picker / EyeDropper, с дедупликацией, FIFO-вытеснением и persistance в `localStorage` (см. раздел 3 → *Recent Colors*).
*   **Custom color picker:** Кнопка с конической градиентной заливкой (`conic-gradient(from 0deg, #ff4757, #ff9f43, #ffd32a, #2ed573, #22d3ee, #1e90ff, #e879f9, #ff4757)`) — открывает поповер `ColorPicker.tsx` с двумя вкладками:
    *   **Pick:** SV-квадрат + Hue-полоса + hex-инпут (ручной подбор одного цвета). При подтверждении (`Confirm`, `Enter`) цвет добавляется в Recent Colors; инструмент переключается на карандаш только если до этого был активен ластик — заливка/штамп остаются активными. Если активный цвет находится вне базовой палитры, кнопка показывает этот цвет вместо градиента (индикатор «работаем кастомным цветом»).
    *   **Generate:** ИИ-генерация палитры через Colormind API (`src/utils/colormindApi.ts`, проксируется локальным `server.ts`) — кнопки 2/3/5 цветов, сгенерированные цвета — квадратные свотчи, кликом можно исключить/включить каждый обратно в выбор (состояние выбора — рамка + тень, без иконки поверх цвета). После первой генерации появляется кнопка **Regenerate** (иконка обновления) — перегенерирует палитру тем же количеством цветов без повторного клика по 2/3/5. Под каждым свотчем — ряд из двух всегда видимых мини-кнопок: замо́к (лочит/анлочит цвет: закреплённые слоты Colormind возвращает без изменений и подбирает остальные цвета под них при следующей генерации/Regenerate) и пипетка, открывающая компактный поповер со своим SV-квадратом/Hue/hex (уменьшенная копия вкладки Pick) и явной кнопкой **Apply** — применяет цвет, закрепляет его (лочит) и помечает выбранным; закрывается по Apply, клику вне поповера или Escape (независимо от всего ColorPicker, не закрывая его). Кнопка **Replace Palette** **заменяет** всю основную палитру (`palette` в `App.tsx`, персистится как `app:palette`) выбранными цветами — не добавляет к существующим. Если выбрано меньше 5 цветов (генерация «2» или «3»), оставшиеся до 5 слотов добиваются базовыми цветами (`#ffffff`, `#ff4757`, `#000000`), чтобы палитра не была слишком скудной для рисования.
*   **EyeDropper:** Кнопка-пипетка, появляется условно (только при наличии `'EyeDropper' in window`). Вызывает нативный селектор пикселя; выбранный цвет добавляется в Recent Colors; инструмент переключается на карандаш только если до этого был активен ластик. Отмена обрабатывается беззвучно.
*   **Инструмент карандаш:** Кнопка `Pencil` — первая в панели инструментов. Клик всегда переводит `activeTool` в `'pencil'`; подсвечивается активной, когда карандаш выбран. Шорткат `B`.
*   **Инструмент ластик:** Кнопка ⌫, переключение между режимами карандаша и ластика. Поскольку ластик не использует цвет, выбор любого цвета (палитра, custom или recent) выводит из ластика обратно в карандаш. Если же был активен инструмент, использующий цвет (заливка, штамп), выбор цвета его не сбрасывает — меняется только активный цвет. Шорткат `E` (тоггл, как и клик).
*   **Инструмент заливка:** Кнопка `PaintBucket` между ластиком и штампом. Повторный клик возвращает в режим карандаша (см. раздел 3 → *Заливка (Flood Fill)*). Шорткат `G` (только silyanka; тоггл).
*   **Инструмент штамп:** Кнопка `Stamp` между заливкой и Mirror Mode. Повторный клик (или выбор любого другого инструмента) возвращает в режим карандаша и сбрасывает захваченный узор (см. раздел 3 → *Штамп (Stamp)*). Пока узор загружен и штамп активен, на углу самой кнопки `Stamp` появляется маленькая кнопка-бейдж переключения точки привязки (`ArrowUpToLine`/`ArrowDownToLine`, top/bottom) — намеренно не отдельная кнопка в общем ряду, а визуально дочерний элемент `Stamp` (`.tool-btn-group`). Шорткат `S` (только silyanka; тоггл — Shift для точки привязки штампа не конфликтует, это отдельная клавиша).
*   **Mirror Mode:** Кнопка-переключатель `FlipHorizontal` для включения/выключения симметричного рисования (см. раздел 3 → *Зеркальный режим*). Расположена сразу после инструмента штампа. При включённом режиме выводит статус в aria-pressed. Шорткат `M` (тоггл, работает в обеих техниках).
*   **Клавиатурные шорткаты инструментов:** Однобуквенные, Photoshop-style — `B` Pencil, `E` Eraser, `G` Flood Fill, `S` Stamp, `M` Mirror Mode (обрабатываются в `App.tsx` вместе с Undo/Redo/Escape/Alt/Shift). Не срабатывают при зажатых `Ctrl/Cmd/Alt`, при авто-повторе (`e.repeat`, зажатая клавиша) и когда фокус находится в поле ввода (`INPUT`/`TEXTAREA`/`contentEditable` — например, при редактировании числового значения Stepper'а). Смена инструмента через шорткат в silyanka идёт через ту же обёртку `setSilyankaTool`, что и клик по кнопке, — сбрасывает захваченный штамп при уходе с него.
*   **Контролы сетки (основная полоса):** Группы ± кнопок для изменения:
    *   **Width** (количество колонн, минимум 1, без верхнего предела): при Mirror Mode изменяет на ±2 (по колонке с каждой стороны). При отключённом режиме добавляет справа. Прямой ввод числа в поле (Stepper `onSet` → `setWidthAbsolute`) при Mirror Mode тоже симметричен: нечётная разница с текущей шириной округляется до чётной, чтобы сохранить центровку рисунка.
    *   **Height** (количество рядов, минимум 1, без верхнего предела): добавляет/удаляет рядов снизу.
    *   **Top Edge** (3–10) и **Bottom Edge** (3–10): глобальные параметры пролётов. Каждый имеет кнопку Reset (↻) для возврата к дефолту и очистки соответствующих per-row override'ов.
    Изменения пересчитываются мгновенно. При уменьшении высоты удаляются подвески и декор-полосы на исчезнувших рядах.
*   **Overflow меню** (при узком экране): кнопка MoreHorizontal (⋯) раскрывает панель с повторением Width, Height, Top Edge, Bottom Edge (со своими Reset кнопками).
*   **Zoom контролы:** Кнопки ± для изменения масштаба (25%–300%, шаг 10%) с отображением текущего значения в процентах.
*   **Управление историей:** Кнопки ↩ (Undo) и ↪ (Redo) с блокировкой при пустых стеках. Сочетания клавиш: `Ctrl+Z` undo, `Ctrl+Y` / `Ctrl+Shift+Z` redo (работают с `Cmd` на macOS).
*   **Очистка (CLEAR):** Кнопка для полной очистки Design Map **и** узора (`colorMap`) на всех размещённых подвесках — сами подвески (размещения) не удаляются, удаляется только покраска их бусин. Операция пишет единый снимок в общую историю (см. ниже) и может быть отменена через Undo.
*   **Библиотека подвесок:** Кнопка `Layers` для открытия/закрытия боковой панели с шаблонами подвесок и управлением их размещением.

### **Разметка и навигация**
*   **Интеллектуальные линейки (`CanvasRulers`)** ([src/components/Editor/CanvasRulers/CanvasRulers.tsx](components/Editor/CanvasRulers/CanvasRulers.tsx)): Автоматически нумеруют ряды и колонны узловых точек с помощью `<text>` элементов SVG. Линейка по X (сверху) привязывается ко второму ряду узлов (r=1), чтобы корректно учитывать горизонтальное смещение нечётных рядов. Линейка по Y (слева) показывает номера рядов из первого (чётного) столбца.
    *   **Per-row Span Control (левое поле):** На левом краю между двумя соседними рядами узлов отрисовываются ± кнопки и счётчик для управления per-row span override'ом (см. раздел 2). Кнопки имеют цветовую кодировку (амбер для нижних пролётов, синий для верхних). При нажатии — мгновенный пересчёт сетки. Дополнительно: строка с ключом `-1` (верхняя цепочка) располагается выше первого ряда и использует общий механизм `rowSpanOverrides`/`onRowSpanChange`. Строка с ключом `-2` (нижняя цепочка) — ниже последнего ряда, отображается только при включённом `BottomEdgeDecor.enabled`, и, в отличие от остальных строк, её ± кнопки управляют `bottomEdgeDecor.span` напрямую (`onBottomEdgeSpanChange`), не затрагивая `rowSpanOverrides`.
    *   **Decor Control (правое поле):** На правом краю выводятся ± кнопки и счётчик для управления числом декор-рядов каждой полосы (см. раздел 3 → *Промежуточный декор*). Диапазон: 0 (полоса отсутствует) ... 10. При клике счётчик показывает текущее число или 0 для неактивной полосы.
    *   **Mirror Axis:** При включённом Mirror Mode отрисовывается вертикальная пунктирная линия по центру (`x = maxX / 2`, где `maxX` — максимальная координата чётного ряда).
*   **Центрирование:** ViewBox SVG считается автоматически по максимальным координатам элементов с учётом технологических отступов (`offsetX = 120px`, `offsetY = 50px`; переход из px в логических координатах происходит в генераторе).

## 4. Аналитика и статистика (Total Count)

Компонент `CanvasStats` играет роль калькулятора материалов в реальном времени:
*   **Total Bead Count:** Общее количество всех бисерин (NODE + SPAN) в текущем проекте **плюс** бисерины всех подвесок (`pendantPlacements`), у которых есть и валидный шаблон, и живая нода-якорь на нижнем ряду (та же проверка занятости, что и в `PendantLayer`). Любое изменение параметров сетки, покраски или размещения подвесок мгновенно обновляет это число (`CanvasView`, `useMemo` над `beads`/`pendantPlacements`).
*   **Color Specification:** Модуль статистики анализирует Design Map и выводит список всех задействованных цветов с указанием точного количества бисерин для каждого из них — включая бисерины подвесок (цвет из `placement.colorMap[index]` или дефолтный цвет типа бусины). Незакрашенные бисерины учитываются по своему цвету по умолчанию (см. раздел 5). Это готовая спецификация для закупки материала.

### **Экспорт схемы в PNG**

Кнопка «Download PNG» (в `CanvasView`, правый нижний угол; стиль консистентен с `.color-picker__confirm`) сохраняет текущую схему как растровое изображение. Весь текст в экспортируемом изображении — на английском. Логика — в [src/utils/exportScheme.ts](utils/exportScheme.ts):

*   **Источник:** живой `<svg>` холста клонируется — это гарантирует попадание в файл всех слоёв (бусины, подвески, нумерация рядов/столбцов, ось зеркала) ровно в том виде, как на экране.
*   **Очистка:** из клона удаляются интерактивные и служебные элементы — span-контролы целиком (кнопки `±` и числа пролётов), хитбоксы, подсветка наведения, drop-таргеты и кнопки удаления подвесок.
*   **Кадрирование:** экранный `viewBox` отбрасывается; реальные границы содержимого замеряются через `getBBox()` (клон временно цепляется за пределами экрана), либо берутся из `options.contentBounds`, если вызывающая сторона передала его явно (так делает CrossWeave — см. раздел 6 → *Экспорт PNG*). Без `contentBounds` это исключает срез линеек, которые рисуются в отрицательных координатах относительно группы трансформации. Вокруг добавляется поле `CONTENT_PADDING`. Через `options.extraStripSelector` можно удалить из клона дополнительные элементы перед экспортом (сверх `STRIP_SELECTORS`) — CrossWeave прячет так незакрашенные бисерины.
*   **Фон:** рисуется фоновый прямоугольник цвета поля холста (`#0f172a`).
*   **Стили:** оформление, заданное CSS-классами (обводка бусин, шрифт нумерации), вшивается в SVG элементом `<style>`, т.к. внешний CSS при сериализации теряется.
*   **Легенда материалов:** под сеткой дорисовывается SVG-блок с заголовком «Materials — total beads: N», списком цветов и количеством бисерин для каждого.
*   **Растеризация:** SVG → `<canvas>` с множителем качества ×2 (резкость под печать) → PNG-файл `silyanka-scheme.png`. Зум на экспорт не влияет — используется натуральный размер.

## 6. Техника «Крестик» (CrossWeave)

Независимый MVP-режим ([useCrossWeaveProject.ts](hooks/useCrossWeaveProject.ts), [CrossWeaveCanvasView.tsx](components/Editor/CanvasView/CrossWeaveCanvasView.tsx)): простая прямоугольная сетка овальных бисерин (`bead-r-c`), без node/span/decor/pendant-концепций силянки — только карандаш, ластик и Mirror Mode. Этот раздел документирует пока только Mirror Mode; остальная функциональность техники в спеке не описана.

### *Зеркальный режим (Mirror Mode)*
Включается кнопкой `FlipHorizontal` в хедере (сразу после ластика, аналог силяночной кнопки — см. раздел 3 → *Зеркальный режим*). Состояние персистентно в `localStorage` (ключ `crossWeave:mirrorMode`).

**Геометрия отражения** ([src/utils/crossWeaveMirror.ts](utils/crossWeaveMirror.ts)): у CrossWeave один тип бисерины, поэтому формула проще, чем `mirrorBeadId` у силянки. Ориентация ряда чередуется по чётности `r`, а не задана типом бисерины:
*   **Нечётный ряд** (горизонтальный, не смещён, `rawWidth` бисерин): `col c ↔ col (rawWidth-1-c)`.
*   **Чётный ряд** (вертикальный, смещён на `pitchX/2`, на одну бисерину короче — `rawWidth-1` бисерин): `col c ↔ col (rawWidth-2-c)`.

`rawWidth` — сырая ширина генератора (`logicalWidth + 1`, см. комментарий `toRawDimensions` в `useCrossWeaveProject.ts`), а не то число колонн, что показывает хедер/линейка.

**Покраска:** клик/мазок карандашом или ластиком закрашивает бисерину и её зеркальную пару через `applyPaint` в `CrossWeaveCanvasView.tsx` (аналог `applyPaint` силянки) — оба изменения проходят через один `paintBead`, поэтому попадают в один шаг истории Undo/Redo неявно (как и в обычном мазке).

**Ось симметрии:** при включённом режиме отрисовывается вертикальная линия (класс `canvas__mirror-axis`, переиспользован из силянки) по центру самого широкого (горизонтального, нечётного) ряда: `x = maxX / 2`.

**Изменение ширины в Mirror Mode:** как и у силянки, кнопки Width ± меняют ширину на **±2** (по колонке с каждой стороны), а прямой ввод числа округляет нечётную разницу до чётной — рисунок остаётся по центру. Design Map пересчитывается функцией `shiftCrossWeaveDesignMapColumns(designMap, delta, newRawWidth)` из того же файла: сдвигает `col` каждой бисерины на `delta`, бисерины за границей сетки отбрасываются. Пересчёт идёт через `remapDesignMap` и пишется в общую историю (Undo/Redo).

### *Экспорт PNG*
Переиспользует `exportSchemeToPng` из [exportScheme.ts](utils/exportScheme.ts) (см. раздел 4 → *Экспорт схемы в PNG*) через необязательный `options: ExportSchemeOptions`, но с двумя отличиями от силянки:
*   **Незакрашенные бисерины скрыты:** `options.extraStripSelector = '.bead--empty'` удаляет их из клона SVG перед растеризацией — в PNG остаются только закрашенные бисерины (класс вешается в `CrossWeaveBeadView.tsx`, `isEmpty = !color`). Применяется только когда закрашена хотя бы одна бисерина — на пустом холсте фолбэк на обычный вид всей сетки.
*   **Кадрирование не по `getBBox()` всего SVG, а по явным границам** (`options.contentBounds`, считаются в `CrossWeaveCanvasView.tsx` как `paintedBounds`): правый/нижний край — впритык к последней закрашенной бисерине (`x/y + beadMajorRadius`); левый/верхний — от 0, **не** обрезается. Причина асимметрии: `getBBox()` тут не годится в принципе — `CrossWeaveRulers` рисует подписи для *всех* рядов/колонок грида независимо от закраски, поэтому измерение bbox всего клона всегда растягивалось бы на весь холст. А сама линейка приклеена к левому/верхнему краю сетки, а не к началу узора, поэтому её label'ы всегда видны целиком, но обрезка слева/сверху для узора, начинающегося не с первой строки/столбца, не работает (остаётся пустое поле с номерами) — известное ограничение.
