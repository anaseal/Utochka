# Документация проекта: Graphic Editor for Silyanka Design

Проект представляет собой специализированный параметрический редактор для создания схем традиционных бисерных украшений (силянок). Система имитирует физику и геометрию сетчатого плетения, позволяя дизайнеру работать с логической структурой изделия, а не с простыми пикселями.

## 1. Основные сущности системы

### **Bead (Бисерина)**
Фундаментальный объект данных. Система разделяет их на две функциональные группы:
*   **NODE (Узел):** Ключевые точки пересечения нитей. Именно они определяют каркас изделия.
*   **SPAN (Соединительная):** Бисерины, заполняющие пространство между узлами на нити.

### **Grid Structure (Сетка)**
Математическая модель, построенная на чередовании рядов узлов.
*   **Четный ряд:** Базовая горизонтальная линия узловых точек.
*   **Нечетный ряд:** Ряд, имеющий горизонтальное смещение на 50% шага относительно четного ряда, что создает ромбовидную ячейку.

### **Design Map (Карта дизайна)**
Реестр состояний (`Record<string, string>`, ID бисерины → цвет), хранящий информацию о цвете каждой закрашенной бисерины. Карта полностью отделена от геометрии: изменение `width`, `height`, span-параметров пересчитывает координаты, но не затрагивает раскраску — цвет сохраняется до тех пор, пока существует бисерина с тем же ID.

## 2. Геометрия и логика плетения

### **Система рядов и колонн**
В отличие от таблиц, здесь расчет ведется по узловым точкам (NODE):
*   **Колонна (Column):** Вертикальная ось узлов. Параметр `width` определяет количество таких осей.
*   **Ряд (Row):** Горизонтальная ось узлов. Параметр `height` определяет количество «этажей» из ромбов. Один полный ромб по вертикали формируется за счёт прохода через три уровня узлов (Верх–Центр–Низ); полное число рядов узлов равно `2 * height + 1`.
*   **Самый верхний ряд (Top Edge):** Особая технологическая цепочка бисерин, соединяющая узлы самого первого ряда (r=0) строго по горизонтали. Она служит «кромкой» изделия.

### **Параметры ромба (Span Control)**
Система позволяет создавать асимметричные ячейки. Span-параметры можно менять как глобально, так и для конкретного ряда:
*   **Top Span (плечи):** Количество бисерин в верхних гранях ромба. Диапазон 3–10, регулируется кнопками ± «Top Edge» в хедере.
*   **Bottom Span (ножки):** Количество бисерин в нижних гранях ромба. Диапазон 3–10, регулируется кнопками ± «Bottom Edge» в хедере.
*   **Per-row Override:** Для каждого span-ряда (промежуток между двумя соседними рядами узлов) на левом поле канваса отрисовываются собственные ± кнопки и счётчик. Они переопределяют глобальные Top/Bottom Edge только для этого ряда; диапазон тот же — 3–10. Цветовая кодировка:
    *   **Нижние пролёты** (r чётный, ножки) — амбер.
    *   **Верхние пролёты** (r нечётный, плечи) — синий.
    *   Если ряд переопределён локально, его счётчик выделяется амбер-цветом, чтобы отличить от наследуемого значения.
    Переопределения хранятся отдельно в реестре `rowSpanOverrides` и не сбрасываются при последующем изменении глобальных Top/Bottom Edge.
*   **Compression:** Вертикальное сжатие (`verticalCompression = 0.25`) имитирует реальную форму бисера, который под собственным весом стремится принять форму приплюснутого овала, а не идеального круга.

### **Адаптивный шаг сетки**
Геометрические шаги рассчитываются на лету и могут не совпадать с базовым `spacing`:
*   **Горизонтальный шаг (`stepX`):** Берётся как максимум из `spacing * horizontalStepMultiplier` и `(internalTop + 1) * minBeadPitch`, где `internalTop = topSpan - 2`, а `minBeadPitch ≈ 14px` — минимальный шаг без перекрытия (`spanRadius * 2 + 2`). Это автоматически раздвигает колонны, если верхняя кромка содержит слишком много бисерин, чтобы поместиться в базовое расстояние.
*   **Вертикальный шаг (`getYStep`):** Считается индивидуально для каждого ряда `r` как `(getInternalCount(r) + 1) * spacing * verticalCompression`. Поскольку `getInternalCount(r)` зависит от фактического span на ряду (с учётом per-row override), высота строки динамическая.

## 3. Функциональные возможности

### **Динамический генератор (`generator.ts`)**
При любом изменении параметров (ширина, высота, глобальные span, per-row overrides) система мгновенно пересчитывает координаты каждой из сотен бисерин и заново выстраивает связи между узлами, формируя целостное полотно. Алгоритм работает в два прохода: сначала формируется сетка узлов (`nodeGrid`), затем по ней интерполируются SPAN-бисерины — вдоль верхней кромки (r=0) и вдоль диагональных граней ромбов.

### **Промежуточный декор (Decor Bands)**
Помимо подвесок (концевой декор), силянка поддерживает **промежуточный декор** — горизонтальные полосы бисерин, вставляемые *между рядами узлов*.
*   **Модель:** Реестр `decorBands` (`Record<number, number>`, ключ — узловой ряд `r`, значение — число декор-рядов полосы `N`). Полоса с ключом `r` располагается в промежутке между узловым рядом `r` и `r+1`. Хранится отдельно, переживает перезагрузку (`localStorage`, ключ `silyanka:decorBands`).
*   **Геометрия:** Полоса состоит из `N` горизонтальных рядов бисерин, повторяющих разметку (ширину и смещение) узлового ряда `r`. Шаг между декор-рядами равен внутреннему шагу пролётов (`spacing · verticalCompression`). Полоса **раздвигает полотно по вертикали**: узловой ряд `r+1` и всё ниже опускаются на `N · шаг`; число колонн не меняется. Нижний декор-ряд полосы становится «виртуальным рядом `r`» — диагональные грани ромбов к ряду `r+1` строятся от него. Декор-бисерины имеют тип `SPAN` и ID вида `decor-{r}-{k}-{c}`, поэтому красятся карандашом/ластиком и учитываются в `CanvasStats` как обычные пролёты.
*   **Управление:** У правой кромки полотна `CanvasRulers` рисует ± контрол на каждый узловой ряд `r` (0…2·height−1). Диапазон `N` — 1–10; значение 0 означает отсутствие полосы. Верхняя горизонтальная грань (r=−1) декор не поддерживает.
*   **Mirror Mode:** Покраска декор-бисерины зеркалит парную по логическому зеркалу колонн (как у NODE).
*   **Изменение сетки:** Ключи `decorBands` привязаны к рядам, а не колоннам, — смена `width` их не затрагивает. При уменьшении `height` полосы на исчезнувших рядах удаляются.

### **Инструменты редактирования**

#### *Карандаш (Pencil)*
Базовый режим. Закрашивание происходит «потоком»: достаточно зажать кнопку мыши и провести курсором по бисеринам — все, через которые проходит указатель, получают активный цвет. Запись в Design Map: `designMap[id] = activeColor`.

#### *Ластик (Eraser)*
Активируется кнопкой ⌫ в хедере. В этом режиме клик по бисерине удаляет её из Design Map (`delete designMap[id]`), возвращая в незакрашенное состояние по умолчанию. Любой клик по цвету палитры или по custom-color picker автоматически возвращает инструмент в режим карандаша.

#### *Custom Color Picker*
Кнопка с конической градиентной заливкой в палитре открывает скрытый системный `<input type="color">`. Выбранный цвет становится активным и автоматически включает карандаш. Если активный цвет не входит в фиксированную палитру, кнопка показывает этот цвет вместо градиента — это служит индикатором «работаем кастомным цветом».

#### *EyeDropper («Пипетка»)*
Доступна только в браузерах, поддерживающих нативный `EyeDropper API` (`'EyeDropper' in window`). Кнопка отрисовывается условно. Клик вызывает системный селектор пикселя — выбранный цвет (`sRGBHex`) подставляется как активный, инструмент переключается в карандаш. Отмена выбора обрабатывается беззвучно.

#### *Recent Colors*
Динамическая секция из 4 inline-слотов в палитре (между базовыми цветами и Custom Picker). Любой цвет, выбранный через Custom Color Picker, EyeDropper или клик по уже существующему recent-слоту, попадает в начало списка; дубликаты дедуплицируются (повторный выбор того же цвета не сдвигает порядок и не создаёт второй слот); при превышении лимита самый старый вытесняется (FIFO). Цвета из базовой палитры **не** записываются в recent — это исключает засорение списка. Список переживает перезагрузку страницы: сериализуется в `localStorage` под ключом `silyanka:recentColors`, при загрузке валидируется (массив hex-строк формата `#RRGGBB`, длина обрезается до лимита; невалидные данные молча игнорируются). Пустые слоты отрисовываются как круги с пунктирной рамкой и не реагируют на клики (`pointer-events: none`).

#### *Hitbox-адаптация*
Для удобства работы с мелкими элементами активная зона клика (`hitboxRadius = 11px`) значительно превышает визуальный размер бисерины (6.5px для NODE, 6.0px для SPAN).

#### *Зеркальный режим (Mirror Mode)*
Режим симметричного рисования, включается кнопкой в хедере (иконка `FlipHorizontal`). При активации на канвасе отрисовывается вертикальная пунктирная ось симметрии по центру (`x = (width-1)·stepX/2`). Любой клик или мазок по бисерине автоматически закрашивает и её зеркальную пару — оба изменения попадают в Design Map за **один** шаг истории (один Undo отменяет пару целиком). Работает и с ластиком: стирание зеркалит так же.

Логика зеркала различается для двух типов бисерин:
*   **NODE — логическое зеркало:** `col c ↔ col (width-1-c)` для всех рядов. Так каждая колонка получает пару, включая правую кромку нечётных рядов. Из-за того, что нечётные ряды смещены на `stepX/2`, зеркальные пары в нечётных рядах визуально не равноудалены от оси — это реальная геометрия плетения силянки.
*   **SPAN — геометрическое зеркало:** пролёты отражаются по фактической геометрии с обменом сторон `left↔right`; индексы бисерин внутри верхней кромки переворачиваются. Логическое зеркало для пролётов невозможно — оно порождало бы грани, которых нет в графе силянки.

Состояние режима сохраняется в `localStorage` (ключ `silyanka:mirrorMode`). При `width ≤ 1` ось не отрисовывается и зеркалирование не выполняется.

**Изменение ширины в Mirror Mode:** при включённом режиме кнопки Width ± меняют ширину на 1, но Design Map автоматически пересчитывается — рисунок сдвигается к новой оси, чтобы остаться по центру (см. раздел 3 → *Контролы сетки*). Пересчёт записывается в историю и отменяется через Undo.

> **Будущее улучшение:** hover-предпросмотр зеркальной бисерины (как «призрак» кисти в Photoshop/Procreate) — при наведении на одну сторону подсвечивать парную бисерину на другой. Пока не реализовано.

#### *Zoom System*
Масштабирование холста от 25% до 300% с шагом 10%. Реализовано CSS-трансформацией (`transform: scale(var(--canvas-zoom))`), поэтому SVG-границы остаются векторно-чёткими на любом уровне зума.

### **История действий (Undo/Redo)**
Система записывает историю на уровне мазков, а не отдельных кликов:
*   **Запись в историю:** Состояние Design Map фиксируется в момент начала мазка (`startDrawing`). При завершении мазка (`stopDrawing`), если карта изменилась, сохранённый снимок добавляется в стек `past`. Операция `clearAll` тоже добавляет снимок в историю перед очисткой, но только если карта была непустой — холостого снимка пустоты не возникает.
*   **Стек истории:** Хранит до 30 снимков (`MAX_HISTORY = 30`). При переполнении самый старый снимок удаляется.
*   **Redo-стек:** Стек `future` сбрасывается при каждом новом мазке или очистке. Undo перемещает текущее состояние в `future`, Redo — возвращает обратно.
*   **Управление:** Кнопки ↩ / ↪ в хедере (с блокировкой при пустых стеках). Клавиатурные сокращения работают как с `Ctrl`, так и с `Cmd` (macOS): `Ctrl/Cmd+Z` — отменить, `Ctrl/Cmd+Y` или `Ctrl/Cmd+Shift+Z` — повторить.

### **Интерфейс управления (Header)**
Фиксированная панель инструментов в верхней части экрана:
*   **Палитра цветов:** 5 хардкод-цветов (`#ff4757`, `#ffd32a`, `#22d3ee`, `#e879f9`, `#ffffff`). Клик по любому цвету активирует карандаш и устанавливает этот цвет как активный. Клики по базовой палитре **не** записываются в Recent Colors.
*   **Recent Colors:** 4 inline-слота сразу после базовой палитры. Накапливают цвета, выбранные через Custom Picker / EyeDropper, с дедупликацией, FIFO-вытеснением и persistance в `localStorage` (см. раздел 3 → *Recent Colors*).
*   **Custom color picker:** Кнопка с конической градиентной заливкой — открывает системный выбор цвета.
*   **EyeDropper:** Кнопка-пипетка, появляется условно (только при наличии `EyeDropper API`).
*   **Инструмент ластик:** Переключение между режимами карандаша и ластика. Выбор любого цвета (палитра или custom) возвращает в режим карандаша.
*   **Mirror Mode:** Кнопка-переключатель симметричного рисования (см. раздел 3 → *Зеркальный режим*). Расположена сразу после ластика.
*   **Контролы сетки:** Группы ± кнопок для изменения **Width** (количество колонн, минимум 1, без верхнего предела), **Height** (количество рядов, минимум 1, без верхнего предела), **Top Edge** и **Bottom Edge** (3–10). Изменения пересчитываются мгновенно. Если **Mirror Mode выключен**, новая колонка добавляется справа. Если **включён** — Design Map пересчитывается так, чтобы рисунок остался по центру относительно оси симметрии (бисерины сдвигаются к новой оси; вышедшие за сетку — отбрасываются).
*   **Zoom контролы:** Кнопки ± для изменения масштаба (25%–300%, шаг 10%) с отображением текущего значения в процентах.
*   **Управление историей:** Кнопки ↩ (Undo) и ↪ (Redo) с блокировкой при пустых стеках.
*   **Очистка (CLEAR):** Кнопка для полной очистки Design Map. Операция записывается в историю и может быть отменена через Undo.

### **Разметка и навигация**
*   **Интеллектуальные линейки (`CanvasRulers`):** Автоматически нумеруют ряды и колонны узловых точек. Линейка по X привязывается ко второму ряду узлов (r=1), чтобы корректно учитывать горизонтальное смещение ромбов. На левом поле тот же компонент отрисовывает per-row span ± контролы (см. раздел 2), на правом — ± контролы промежуточного декора (см. раздел 3 → *Промежуточный декор*).
*   **Центрирование:** ViewBox SVG считается автоматически по максимальным координатам элементов с учётом технологических отступов (`offsetX = 160px`, `offsetY = 50px`).

## 4. Аналитика и статистика (Total Count)

Компонент `CanvasStats` играет роль калькулятора материалов в реальном времени:
*   **Total Bead Count:** Общее количество всех бисерин (NODE + SPAN) в текущем проекте. Любое изменение параметров сетки мгновенно обновляет это число.
*   **Color Specification:** Модуль статистики анализирует Design Map и выводит список всех задействованных цветов с указанием точного количества бисерин для каждого из них. Незакрашенные бисерины учитываются по своему цвету по умолчанию (см. раздел 5). Это готовая спецификация для закупки материала.
