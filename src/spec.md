# Документация проекта: Graphic Editor for Silyanka Design

Проект представляет собой специализированный параметрический редактор для создания схем традиционных бисерных украшений (силянок). Система имитирует физику и геометрию сетчатого плетения, позволяя дизайнеру работать с логической структурой изделия, а не с простыми пикселями.

## 1. Основные сущности системы

### **Bead (Бисерина)**
Фундаментальный объект данных. Система разделяет их на две функциональные группы:
*   **NODE (Узел):** Ключевые точки пересечения нитей. Именно они определяют каркас изделия.
*   **SPAN (Соединительная):** Бисерины, заполняющие пространство между узлами на нити.

### **Grid Structure (Сетка)**
Математическая модель, построенная на чередовании рядов узлов.
*   **Четный ряд:** Базовая горизонтальная линия узловых точек.
*   **Нечетный ряд:** Ряд, имеющий горизонтальное смещение на 50% шага относительно четного ряда, что создает ромбовидную ячейку.

### **Design Map (Карта дизайна)**
Реестр состояний (`Record<string, string>`, ID бисерины → цвет), хранящий информацию о цвете каждой закрашенной бисерины. Карта полностью отделена от геометрии: изменение `width`, `height`, span-параметров пересчитывает координаты, но не затрагивает раскраску — цвет сохраняется до тех пор, пока существует бисерина с тем же ID.

## 2. Геометрия и логика плетения

### **Система рядов и колонн**
В отличие от таблиц, здесь расчет ведется по узловым точкам (NODE):
*   **Колонна (Column):** Вертикальная ось узлов. Параметр `width` (внутреннее состояние, `gridSize.width`) определяет количество узлов в чётном ряду; в нечётном ряду (со сдвигом на полшага) их физически **на один больше** — по одному лишнему узлу с каждого края (`c = -1..width-1`, см. «Диапазон колонок нечётных рядов» в разделе 3), а не на один меньше, как можно было бы ожидать. При этом линейка (`CanvasRulers`) по-прежнему показывает `width-1` «колонок» — два крайних узла нечётного ряда специально исключены из подсчёта отдельным фильтром.
*   **Ряд (Row):** Горизонтальная ось узлов. Параметр `height` (внутреннее состояние, `gridSize.height`) определяет количество «этажей» из ромбов. Один полный ромб по вертикали формируется за счёт прохода через три уровня узлов (Верх–Центр–Низ); полное число рядов узлов равно `2 * height + 1`, из них чётных (промаркированных линейкой) — `height + 1`.
*   **Ширина/высота в хедере vs. внутреннее состояние:** Степперы Width/Height в `Header` показывают и принимают числа, совпадающие с линейкой, а не сырые `gridSize.width/height` — иначе цифра в хедере не совпадала бы с тем, что физически видно на холсте (при `width=30` линейка насчитывает 29 колонок, при `height=8` — 9 рядов). Пересчёт `±1` сделан в одной точке — [src/App.tsx](App.tsx) (`silyankaProps.gridWidth/gridHeight/onSetWidth/onSetHeight`); `generator.ts`, `mirror.ts` и остальная геометрия по-прежнему работают с исходными `gridSize.width/height`. Кнопки ± (`onWidthChange`/`onHeightChange`) меняют `gridSize` напрямую дельтой — сдвиг на константу не требует пересчёта.
*   **Самый верхний ряд (Top Edge):** Особая технологическая цепочка бисерин, соединяющая узлы самого первого ряда (r=0) дугой (не строго по горизонтали — см. «Дуга кромок» в разделе 3). Она служит «кромкой» изделия.
*   **Самый нижний ряд (Bottom Edge):** По умолчанию — просто последний ряд узлов. Независимый от Top/Bottom Edge и per-row override'ов декор `BottomEdgeDecor { enabled, span }` может добавить аналогичную верхней горизонтальную цепочку ниже последнего узлового ряда — см. «Нижняя цепочка (Bottom Chain)» в разделе 3.

### **Параметры ромба (Span Control)**
Система позволяет создавать асимметричные ячейки. Span-параметры можно менять как глобально, так и для конкретного ряда:
*   **Top Span (плечи):** Количество бисерин в верхних гранях ромба. Диапазон 3–10, регулируется кнопками ± «Top Edge» в хедере. При нажатии кнопки Reset (иконка ↻) глобальное значение топ-спана вернётся к дефолту (`beadsInSpan = 3`), а все per-row override для нечётных рядов (плечи, `r % 2 !== 0`) удаляются.
*   **Bottom Span (ножки):** Количество бисерин в нижних гранях ромба. Диапазон 3–10, регулируется кнопками ± «Bottom Edge» в хедере. Reset кнопка вернёт дефолт и очистит per-row override для чётных рядов (ножки, `r % 2 === 0`).
*   **Per-row Override:** Для каждого span-ряда (промежуток между двумя соседними рядами узлов) на левом краю канваса отрисовываются собственные ± кнопки и счётчик. Они переопределяют глобальные Top/Bottom Edge только для этого ряда; диапазон тот же — 3–10. Цветовая кодировка:
    *   **Нижние пролёты** (r чётный, ножки) — амбер.
    *   **Верхние пролёты** (r нечётный, плечи) — синий.
    *   **Верхняя горизонтальная грань** (r=−1) — также синяя (плечи), имеет собственный override.
    *   Если ряд переопределён локально, его счётчик выделяется более ярким цветом, чтобы отличить от наследуемого значения.
    Переопределения хранятся отдельно в реестре `rowSpanOverrides` ([src/App.tsx](App.tsx): `usePersistedState('silyanka:rowSpanOverrides', ...)`) и в общем случае **не сбрасываются** при последующем изменении глобальных Top/Bottom Edge. Исключение: если изменение глобального параметра делает override равным новому дефолту ряда, такой избыточный override удаляется (`pruneRedundantOverrides` в `updateTopSpan`/`updateBottomSpan`) — иначе он «протух» бы и навсегда заморозил ряд, поскольку `resolveSpanCount` отдаёт приоритет override'у. Reset же кнопки специально отсеивают override'ы для соответствующих рядов по чётности.
    Логика преобразования span-параметров в зависимости от чётности ряда централизована в [src/utils/spans.ts](utils/spans.ts): функция `resolveSpanCount(r, topSpan, bottomSpan, rowSpanOverrides)` — возвращает эффективное количество бисерин для пролёта ряда с учётом override'а.
*   **Compression:** Вертикальное сжатие (`verticalCompression = 0.2`) имитирует реальную форму бисера, который под собственным весом стремится принять форму приплюснутого овала, а не идеального круга.

### **Плотность сетки (Spacing)**
Базовое расстояние между узлами `spacing` (px) регулируется в хедере степпером «Spacing» (рядом с Zoom, есть также в overflow-панели для узких экранов). Диапазон **45–85px**, шаг **5px** (`BEAD_THEME.constraints.minSpacing/maxSpacing/spacingStep`, [src/config/theme.ts](config/theme.ts)), дефолт — 65px. Значение хранится в `GridConfig.spacing` ([src/App.tsx](App.tsx): `usePersistedState('silyanka:gridSize', ...)`) и напрямую передаётся в `generateSilyankaGrid` — генератор уже был параметричен по `spacing`, изменение плотности не требует правок геометрии, только масштабирует `stepX`/`getYStep`/`decorRowStep` (см. ниже). ViewBox канваса считается динамически от bounding box бисерин, поэтому корректно подстраивается под любую плотность.

### **Адаптивный шаг сетки**
Геометрические шаги рассчитываются на лету и могут не совпадать с базовым `spacing`:
*   **Горизонтальный шаг (`stepX`):** Берётся как максимум из `spacing * horizontalStepMultiplier`, `(internalTop + 1) * minBeadPitch` и `(internalBottom + 1) * minBeadPitch`, где `internalTop = topEdgeCount - 2` (верхняя кромка, r=-1), `internalBottom = bottomEdgeSpan - 2` (нижняя цепочка, 0 если она выключена), а `minBeadPitch ≈ 14px` — минимальный шаг без перекрытия (`spanRadius * 2 + 2`). Это автоматически раздвигает колонны, если верхняя или нижняя кромка содержит слишком много бисерин, чтобы поместиться в базовое расстояние.
*   **Вертикальный шаг (`getYStep`):** Считается индивидуально для каждого ряда `r` как `(getInternalCount(r) + 1) * spacing * verticalCompression`. Поскольку `getInternalCount(r)` зависит от фактического span на ряду (с учётом per-row override), высота строки динамическая.

## 3. Функциональные возможности

### **Динамический генератор (`generator.ts`)**
При любом изменении параметров (ширина, высота, глобальные span, per-row overrides) система мгновенно пересчитывает координаты каждой из сотен бисерин и заново выстраивает связи между узлами, формируя целостное полотно. Алгоритм работает в два прохода: сначала формируется сетка узлов (`nodeGrid`), затем по ней интерполируются SPAN-бисерины — вдоль верхней кромки (r=0), вдоль диагональных граней ромбов и, при включённом `BottomEdgeDecor.enabled`, вдоль нижней кромки.

#### **Дуга кромок (Top/Bottom Edge Arc)**
Верхняя кромка (r=0) и нижняя цепочка (при включении) — не плоская «полка» с постоянным Y-офсетом, а дуга: каждая внутренняя бисерина смещается от прямой между узлами по `sin`-профилю. Для бисерины `i` (позиция `1..count`), `t = i / (count + 1)`:

`y += direction · edgeArcHeight · sin(π · t)`

где `direction = -1` для верхней кромки (дуга наружу, вверх от полотна) и `direction = +1` для нижней. `edgeArcHeight = 13` ([src/config/theme.ts](config/theme.ts)). У узлов (`t → 0` и `t → 1`) смещение стремится к нулю, у середины пролёта (`t = 0.5`) — максимально; это не даёт бисеринам наложиться друг на друга при большом числе бисерин в кромке (в отличие от прежнего константного офсета). Реализовано функцией `generateArcSpan` в [src/utils/generator.ts](utils/generator.ts), отдельно от `generateSpan` — последняя по-прежнему используется для диагональных граней ромбов, без дуги (`yOffset = 0`).

#### **Диапазон колонок нечётных рядов**
Нечётные (сдвинутые на `stepX/2`) ряды занимают колонки `c = -1..width-1` — на одну шире с каждой стороны, чем чётные (`c = 0..width-1`), а не `0..width-2`, как можно было бы ожидать из «на один узел меньше, чем в чётном ряду» (раздел 2). Без этого расширения у крайнего узла чётного ряда (`c=0` или `c=width-1`) снизу/сверху был бы только один сосед вместо двух — единственная диагональ вместо полного зигзага, причём именно там, где чётный ряд — самый первый (`r=0`) или самый последний (`r=2·height`) и компенсировать нечем (у промежуточных чётных рядов недостающую сторону всегда закрывает диагональ от соседнего ряда с другой стороны). `minC(r) = r % 2 !== 0 ? -1 : 0`, `maxC = width - 1` — единая формула в [src/utils/generator.ts](utils/generator.ts) (`nodeAt(r, c)` вместо прямой индексации массива `nodeGrid[r][c]`, т.к. индекс в массиве — `c - minC(r)`). Существующий `node`/`vertEdge` — никакого отдельного вида бисерины: `mirror.ts` (`mirrorCol`) и `regrid.ts` используют ту же формулу зеркала/сдвига колонки, что и раньше, только с расширенными границами; `stamp.ts`/`floodFill.ts` не менялись вовсе — они уже работали по факту существования бисерины (`ctx.beadIds.has(...)`), а не по диапазону.
*   **Линейка колонок** (`CanvasRulers`) по-прежнему показывает то же число колонок, что и раньше: два новых крайних узла (`c=-1` и `c=width-1` в ряду `r=1`) исключены из подсчёта отдельным фильтром по границам `0..width-2`, без изменения самой логики подсчёта.

#### **Нижняя цепочка (Bottom Chain)**
Аналог верхней горизонтальной кромки, добавляемый **ниже** последнего узлового ряда (`r = 2·height`), той же дугой. В отличие от Top/Bottom Edge и per-row override'ов, полностью независима от них: хранится в отдельном персистентном стейте `BottomEdgeDecor { enabled: boolean; span: number }` ([src/types/bead.ts](types/bead.ts); `usePersistedState('silyanka:bottomEdgeDecor', ...)` в [src/App.tsx](App.tsx)) — не читает и не пишет `rowSpanOverrides`/`topSpan`/`bottomSpan`. `span` (диапазон 3–10, дефолт 3) задаёт число бисерин напрямую: `internalBottom = max(0, span - 2)`. ID бисерин — `span-edge-bottom-link-${c}-bead-${i}` (не содержат номер ряда, поэтому стабильны при изменении `height`). Поддержка:
*   **Flood Fill:** нижняя цепочка подключена к графу смежности (`floodFill.ts`) через конечные узлы `node-{lastR}-c` и `node-{lastR}-{c+1}`, как и верхняя кромка.
*   **Mirror Mode:** зеркалирование по той же логике, что и у верхней кромки (`mirror.ts`), с `internalBottom = bottomEdgeDecor.span - 2`.
*   **UI:** тумблер «Bottom Chain» — в боковой панели `PendantsSidebar` (первый раздел панели, над Pendants и Decor), включает/выключает `enabled`. Степпер количества бисерин (±, диапазон 3–10) — на канвасе в `CanvasRulers`, в той же позиции под последним рядом узлов; управляет `bottomEdgeDecor.span` напрямую, минуя `rowSpanOverrides` и общий Bottom Edge из хедера.
*   **Взаимоисключение с подвесками:** Bottom Chain и точечные подвески (`pendantPlacements`) не могут быть включены одновременно — тумблер задизейблен, пока на канвасе есть хотя бы одна подвеска (`toggleBottomEdgeEnabled` в [src/App.tsx](App.tsx)); каталог шаблонов подвесок, в свою очередь, задизейблен, пока включён Bottom Chain (`PendantsSidebar.tsx`).

### **Промежуточный декор (Decor Bands)**
Помимо подвесок (концевой декор), силянка поддерживает **промежуточный декор** — горизонтальные полосы бисерин, вставляемые *между рядами узлов*.
*   **Модель:** Реестр `decorBands` ([src/hooks/useSilyankaProject.ts](hooks/useSilyankaProject.ts): `usePersistedState('silyanka:decorBands', ...)`; `Record<number, number>`, ключ — узловой ряд `r`, значение — число декор-рядов полосы `N`). Полоса с ключом `r` располагается в промежутке между узловым рядом `r` и `r+1`. Хранится отдельно, переживает перезагрузку (`localStorage`, ключ `silyanka:decorBands`). Минимальное значение для активной полосы — 1; значение 0 или отсутствие ключа означают отсутствие полосы (диапазон хранения: 1–10 строк).
*   **Геометрия:** Полоса состоит из `N` горизонтальных рядов бисерин, повторяющих разметку (ширину и смещение) узлового ряда `r`. Шаг между декор-рядами равен внутреннему шагу пролётов: `spacing · verticalCompression`. Полоса **раздвигает полотно по вертикали**: узловой ряд `r+1` и всё ниже опускаются на `N · шаг`; число колонн не меняется. Нижний декор-ряд полосы становится «виртуальным рядом `r`» — диагональные грани ромбов к ряду `r+1` строятся от него (координаты граней рассчитываются генератором с учётом расширения). Декор-бисерины имеют тип `SPAN` и ID вида `decor-{r}-{k}-{c}` (где `k` — индекс декор-ряда в полосе, от 0 до N−1), поэтому красятся карандашом/ластиком и учитываются в `CanvasStats` как обычные пролёты.
*   **Управление:** В сайдбаре (раздел «Decor», последний из трёх разделов панели, без переключения вкладок) находится карточка «Band». Её перетаскивают на зазор между рядами на канвасе — во время drag `CanvasRulers` подсвечивает ближайший зазор (розовая полоса); при сбросе полоса добавляется (или удаляется, если уже существует — toggle). Диапазон `N` — 1–10 (`decorDefaults.minRows = 1`, `decorDefaults.maxRows = 10`); после размещения счётчик полосы регулируется кнопками ± в списке «Placed» в том же разделе. Верхняя горизонтальная грань (r=−1) декор не поддерживает. Для вычисления зазоров сайдбар получает проп `rowGaps: {row,midY}[]`, вычисляемый в `App.tsx` из NODE-бусин.
*   **Mirror Mode:** Покраска декор-бисерины зеркалит парную по логическому зеркалу колонн (как у NODE): `col c ↔ col (width-1-c)` для чётных рядов, `col c ↔ col (width-2-c)` для нечётных.
*   **Изменение сетки:** Ключи `decorBands` привязаны к рядам узлов, а не к колоннам, — смена `width` их не затрагивает. При уменьшении `height` полосы на исчезнувших рядах удаляются ([src/hooks/useSilyankaProject.ts](hooks/useSilyankaProject.ts): общий хелпер `applyHeight` — вызывается из `updateDimension('height', ...)` и `setHeightAbsolute` — отфильтровывает `decorBands` так, чтобы остались только ряды `r < 2·newHeight`).

### **Инструменты редактирования**

#### *Ввод: мышь и тач (Pointer Events)*
Весь холст (карандаш, ластик, заливка, штамп, покраска подвесок) слушает Pointer Events (`onPointerDown`/`onPointerEnter`/`onPointerUp`), а не мышиные — единый код для мыши, тача и пера. Ключевой момент для тача: на `pointerdown` каждая бисерина вызывает `releasePointerCapture` на `e.target` (реально захваченный browser'ом элемент — вложенный `<circle>`/`<ellipse>`, а не `<g>`, на котором висит обработчик) — без этого браузер неявно «привязывает» палец к бисерине, на которой было касание, и `onPointerEnter` соседних бисерин не срабатывает при проводке пальцем (рисование линией не работало бы). Вызов `releasePointerCapture` на `e.currentTarget` — частая ошибка: `<g>` capture не держит, и release на нём молча ничего не делает. Область холста (`.canvas__svg`) имеет `touch-action: none` — без этого жест рисования перехватывался бы нативным скроллом/pinch-zoom страницы. Там же `user-select: none` / `-webkit-user-select: none` / `-webkit-touch-callout: none` — без них протяжка пальцем параллельно триггерит нативное выделение текста/содержимого страницы (хэндл выделения и системное меню снизу экрана на мобильном) поверх рисования линией.

**Панорама и pinch-zoom двумя пальцами** ([src/hooks/useTouchPanZoom.ts](hooks/useTouchPanZoom.ts), общий для `CanvasView` и `CrossWeaveCanvasView`): раз `touch-action: none` отключает нативные жесты браузера, листание и pinch-zoom на тач-устройствах реализованы вручную поверх Pointer Events. Один палец — обычное рисование (не тронуто). Как только опускается второй палец, `onPointerDownCapture` на `<main>` (capture-фаза, раньше вложенных `onPointerDown`) гасит событие через `stopPropagation` — второй палец не красит бисерину под собой и не путает драг штампа — и отменяет уже начатый мазок первого пальца (`stopDrawing` + сброс `stampDragRef`/`selectionRect` в `CanvasView`). Пока пальцев два: сведение/разведение меняет zoom (`onZoomChange`, та же непрерывная дельта, что и у `Ctrl`+колесо в `useWheelZoom`), сдвиг средней точки между пальцами скроллит контейнер (`scrollLeft`/`scrollTop` напрямую). Отпускание одного из двух пальцев завершает жест; рисование одним пальцем не возобновляется автоматически — нужно оторвать оба и начать заново.

Новый zoom на каждый `pointermove` считается не инкрементально, а абсолютно — как `gesture.startZoom · (distance(a,b) / gesture.startDist)` от значений `startZoom`/`startDist`, зафиксированных в момент начала жеста (`useTouchPanZoom.ts`), поэтому «удвоения дельты» между двумя `pointermove` одного и того же движения (по одному на каждый палец) не возникает в принципе. `liveZoomRef` тем не менее обновляется сразу же внутри `onPointerMove` (а не только через проп `zoom`, который во время жеста не меняется вовсе) — по другой причине: если поверх уже идущего жеста опускается третий палец, новый под-жест должен стартовать не от устаревшего React-пропа `zoom` (он не обновлялся с начала пинча), а от актуального «живого» значения — иначе масштаб дёрнулся бы обратно к состоянию до начала первого жеста.

#### *Карандаш (Pencil)*
Базовый режим. Закрашивание происходит «потоком»: достаточно зажать кнопку мыши (или прикоснуться пальцем) и провести по бисеринам — все, через которые проходит указатель, получают активный цвет. Запись в Design Map: `designMap[id] = activeColor`. Явно активируется кнопкой `Pencil` в хедере (первая в панели инструментов, перед ластиком) — клик по ней всегда возвращает в режим карандаша; кнопка подсвечивается как активная, когда `activeTool === 'pencil'`.

#### *Ластик (Eraser)*
Активируется кнопкой ⌫ в хедере. В этом режиме клик по бисерине удаляет её из Design Map (`delete designMap[id]`), возвращая в незакрашенное состояние по умолчанию. Поскольку ластик не использует цвет, выбор любого цвета (палитра, custom или recent) выводит из режима ластика обратно в карандаш; если же был активен инструмент, использующий цвет (заливка, штамп), выбор цвета лишь меняет активный цвет и не переключает инструмент.

#### *Заливка (Flood Fill)*
Активируется кнопкой `PaintBucket` в хедере (между ластиком и Mirror Mode). Повторный клик возвращает инструмент в режим карандаша. При активном режиме курсор канваса меняется на `crosshair`. Один клик по бисерине закрашивает всю связную область бисерин **того же эффективного цвета** активным цветом за один шаг истории (Undo отменяет всю заливку разом).

**Алгоритм:** BFS по графу смежности бисерин, построенному из массива `beads`. Связи:
*   **SPAN-цепочки** (верхняя кромка и диагональные грани ромбов): каждый кластер образует линейную цепь `startNode → bead-1 → … → bead-N → endNode`, где конечные узлы определяются из cluster ID (например, `edge-top-link-{c}` → `node-0-{c}` и `node-0-{c+1}`).
*   **Декор-полосы**: вертикальная смежность между рядами `k` и `k+1` одной колонки + горизонтальная смежность между соседними колонками в пределах одного ряда `k`. Декор-полосы **изолированы** от основной сетки (нет рёбер между `decor-*` и `node-*`/`span-edge-*`), поэтому заливка не «перетекает» из полосы в сетку и наоборот.

**Цвет старта:** `designMap[id] ?? defaultColorFor(bead.type)`. Если эффективный цвет стартовой бисерины уже равен активному цвету — заливка не выполняется (нет снимка в историю).

**Mirror Mode:** если режим включён, заливка выполняется также для зеркальной бисерины; оба результата объединяются в один вызов `remapDesignMap`.

#### *Штамп (Stamp)*
Активируется кнопкой `Stamp` в хедере (сразу после Flood Fill). Позволяет выделить рамкой раскрашенный мотив и многократно переносить его в другое место сетки, пока инструмент не переключат.

**Взаимодействие** (реализовано в `CanvasView`, вся геометрия — в [src/utils/stamp.ts](utils/stamp.ts)):
*   **Drag на мыши (сдвиг > 4px, независимо от zoom)** — рисует рамку выделения и по отпусканию кнопки захватывает все бисерины (любого типа — узлы, пролёты, декор, кромки), чей центр (`bead.x/y`, без учёта радиуса) попал внутрь прямоугольника. Координаты рамки считаются через `getScreenCTM()` того `<g>`, что несёт `translate(offsetX, offsetY)`, — сразу в системе координат бисерин. В момент, когда драг распознан (превышен порог), протухший preview предыдущего узора сразу гасится, чтобы не мешать видеть новую рамку. На тач тот же порог (`STAMP_DRAG_THRESHOLD_TOUCH = 10px` вместо `4px` — палец толще и дрожит сильнее курсора) применяется, только пока узор ещё не загружен: рисование рамки на тач работает так же, просто с более щадящим порогом.
*   **Click (без сдвига) на мыши при загруженном узоре** — берёт ближайшую NODE-бисерину к точке клика как целевой якорь и ставит копию узора. Инструмент остаётся активным с тем же узором — повторные клики ставят новые копии, одна запись в историю (Undo) на одну постановку.
*   **Новый drag на мыши поверх уже загруженного узора** заменяет его новым выделением.
*   **Тач при загруженном узоре — отдельная модель (`mode: 'movePreview'` в `stampDragRef`, CanvasView):** наведения без контакта на тач не бывает, поэтому вместо клика/драга касание сразу таскает превью. Палец вниз — сразу считается ближайшая NODE и показывается превью в этой точке; пока палец не отпущен, каждое перемещение пересчитывает ближайшую NODE и двигает превью следом (без порога — драга в обычном смысле в этом состоянии нет); отпускание пальца ставит копию в последней подсвеченной позиции. Побочный эффект: пока узор загружен, тач не может «с одного касания» нарисовать новую рамку выделения (в отличие от мыши) — сначала нужно сбросить узор кнопкой-крестиком (см. Escape/Alt ниже), затем тянуть новую рамку обычным образом.
*   **Живое превью:** пока узор загружен и есть целевая позиция (наведение мышью или таскание пальцем на тач), в целевых бисеринах поверх их текущего цвета рисуется полупрозрачный пунктирный кружок цвета из узора (`bead__preview` в `BeadView`, цвета берутся из `stampPreviewPatch` — полного патча `id → color`, который строит `applyStampPattern` для текущей наведённой позиции) — превью показывает реальный будущий рисунок, а не просто подсвечивает контур бисерины.
*   **Escape** сбрасывает загруженный узор (без выделения ничего не происходит). **Alt** делает то же самое — быстрый способ сразу вернуться в режим выделения новой рамки, не отпуская мышь ради Escape. На тач-экране, где нет клавиатуры, тот же сброс доступен кнопкой-крестиком (`X`, `.tool-btn-group__badge--cancel`) — вторым бейджем на противоположном углу кнопки `Stamp`, рядом с бейджем точки привязки; вызывает общий `cancelStampPattern` в `App.tsx`, используемый также хендлерами Escape/Alt. **Переключение на любой другой инструмент** (карандаш/ластик/заливка/выбор другого цвета через `selectColor`) делает то же самое — обёртка `setSilyankaTool` в `App.tsx` сравнивает старый и новый `activeTool`. Так при каждом новом заходе в штамп нет «залипшего» узора от прошлой сессии, который иначе сразу показывал бы превью и мешал выделить новую область.
*   **Точка привязки (top/bottom):** по умолчанию `targetAnchor` совмещается с верхним рядом захваченного узора (`anchorRow`/`anchorCol`). Пока узор загружен, у кнопки `Stamp` в хедере появляется маленький кнопка-бейдж (`ArrowUpToLine`/`ArrowDownToLine`) на её углу — визуально дочерний тоггл, а не самостоятельный инструмент в общем ряду (`.tool-btn-group` / `.tool-btn-group__badge` в `Header.css`). Клик по бейджу **или** клавиша **Shift** (тап, не удержание — эквивалент клика, слушается в `App.tsx` вместе с Escape/Alt, `e.repeat` отсекает авто-повтор) переключают `stampAnchorEdge` насовсем — на нижний ряд узора (`anchorRowBottom`/`anchorColBottom`, тоже вычисляется в `captureStampPattern`) и обратно. Так можно навести курсор на существующий узел у верхнего края полотна и поставить туда именно *низ* штампа, а верхняя часть узора, «уехавшая» выше ряда 0, естественно обрежется (см. финальную защиту по `beadIds` ниже) — без необходимости заводить курсор за пределы канваса. Оба способа переключения полностью эквивалентны — состояние одно на весь сеанс штампа, а не временный оверрайд на одну постановку (это обсуждалось и было отклонено: удерживать модификатор ради каждого клика неудобно).
*   **Mirror Mode:** после постановки штампа каждая успешно перенесённая бисерина дополнительно прогоняется через `mirrorBeadId` — штамп ведёт себя симметрично карандашу.

**Модель трансляции id** (`translateBeadId(id, dRow, dCol, ctx, anchorRow)`, аналог `mirrorBeadId` для сдвига, а не отражения) — у сетки нет равномерной решётки, поэтому перенос id пересчитывается по правилам, а не сложением чисел:
*   `node-{r}-{c}` → `node-{r+dRow}-{c+dCol+correction}`. `correction` компенсирует то, что нечётные ряды физически сдвинуты на `stepX/2` относительно чётных (`generator.ts` — `isShifted`): при нечётном `dRow` (перенос через ряд другой чётности) бисерины, чья исходная чётность ряда `r` отличается от чётности ряда-якоря узора (`anchorRow`), получают `±1` к колонке — без этого узор рвётся на несвязанные «чётную»/«нечётную» половины, съезжающие друг относительно друга на целый шаг сетки. При чётном `dRow` `correction = 0` всегда. Та же коррекция применяется к диагональным граням (`span-edge-{r}-{c}-...`) и декору (`decor-{r}-{k}-{c}`) — они используют ту же систему координат `(r, c)`, что и NODE.
*   `span-edge-{r}-{c}-{side}-bead-{i}` (диагональные грани ромба) — origin-узел переносится как NODE (включая ту же коррекцию колонки); индекс бисерины внутри грани переиндексируется пропорционально: `t = i/(srcCount+1)`, `i' = round(t·(dstCount+1))`, где `srcCount`/`dstCount` — число бисерин в пролёте исходного/целевого ряда (та же формула, что `getInternalCount` в `generator.ts`). Это покрывает перенос между рядами с разным per-row override.
*   `span-edge-top-link-{c}-bead-{i}` и `span-edge-bottom-link-{c}-bead-{i}` (верхняя/нижняя кромка) — жёстко привязаны к одному ряду (r=0 и r=2·height), поэтому переносятся **только по горизонтали** (`dRow === 0`); при вертикальном сдвиге штампа эти бисерины выпадают из вставки, остальные переносятся как обычно. Это свойство модели, а не баг.
*   `decor-{r}-{k}-{c}` → `decor-{r+dRow}-{k}-{c+dCol+correction}` (та же коррекция колонки, что у NODE), допустимо только если у целевой полосы `decorBands[r+dRow] >= k`.
*   `pendant:*` не поддерживается (отдельная система `colorMap`, вне Design Map).
*   Финальная защита для всех правил: кандидат принимается, только если он реально присутствует в текущем массиве `beads` (`Set<string>` по id) — подстраховка от ошибок в ручной геометрии; частично «свисающая» за край сетки вставка просто теряет недостижимые бисерины, без визуальных артефактов.

`captureStampPattern(ids, beads, designMap)` фиксирует **два** anchor'а — минимальный и максимальный `(row, col)` среди захваченных NODE-бисерин (или среди всех, если узлов в выделении нет): `anchorRow/anchorCol` (верхний ряд выделения) и `anchorRowBottom/anchorColBottom` (нижний ряд) — и список `{id, color}` **только для реально закрашенных бисерин** (`designMap[id] !== undefined`). Незакрашенные бисерины внутри рамки выделения в узор не попадают — штамп переносит рисунок, а не «допечатывает» пустоту поверх того, что уже может быть на месте вставки (иначе `designMap[id]` получил бы буквальное значение цвета-плейсхолдера, что ломает проверку `isEmpty` в `BeadView` и рендерится некорректно). `applyStampPattern(pattern, targetAnchor, ctx, edge = 'top')` выбирает пару anchor по `edge` (`'top'` → `anchorRow/anchorCol`, `'bottom'` → `anchorRowBottom/anchorColBottom`), считает `dRow/dCol` относительно неё и прогоняет каждую запись через `translateBeadId`, отбрасывая недостижимые id; результат — один патч `{id: color}`, применяемый через `remapDesignMap` за один шаг истории.

#### *Custom Color Picker*
Кнопка с конической градиентной заливкой в палитре открывает скрытый системный `<input type="color">`. Выбранный цвет становится активным и автоматически включает карандаш. Если активный цвет не входит в фиксированную палитру, кнопка показывает этот цвет вместо градиента — это служит индикатором «работаем кастомным цветом».

#### *EyeDropper («Пипетка»)*
Доступна только в браузерах, поддерживающих нативный `EyeDropper API` (`'EyeDropper' in window`). Кнопка отрисовывается условно. Клик вызывает системный селектор пикселя — выбранный цвет (`sRGBHex`) подставляется как активный, инструмент переключается в карандаш. Отмена выбора обрабатывается беззвучно.

#### *Recent Colors*
Динамическая секция из 5 слотов в палитре — отдельная строка под базовыми цветами (двухэтажная раскладка `.palette__grid`: сверху 5 базовых цветов, снизу 5 recent-слотов, симметрично). Любой цвет, выбранный через Custom Color Picker, EyeDropper или клик по уже существующему recent-слоту, попадает в начало списка; дубликаты дедуплицируются (повторный выбор того же цвета не сдвигает порядок и не создаёт второй слот); при превышении лимита самый старый вытесняется (FIFO). Цвета из базовой палитры **не** записываются в recent — это исключает засорение списка. Список переживает перезагрузку страницы: сериализуется в `localStorage` под ключом `silyanka:recentColors`, при загрузке валидируется (массив hex-строк формата `#RRGGBB`, длина обрезается до лимита; невалидные данные молча игнорируются). Пустые слоты отрисовываются как круги с пунктирной рамкой и не реагируют на клики (`pointer-events: none`).

#### *Подвески (Pendants)*
Система для добавления декоративных подвесок на нижние узлы сетки:
*   **Шаблоны подвесок** ([src/data/pendantTemplates.ts](data/pendantTemplates.ts)): Предопределённый набор из 9 шаблонов (Drop, Rhombus, Cluster, Fringe, Net, Net with cluster, Drop with cluster, Bugle bead, Loop). Каждый шаблон состоит из массива бусин с указанием смещения (`dx`, `dy`) от якорной ноды, формы (`circle` или `rect`) и типа (`NODE` или `SPAN`).
    *   **Loop** — петля из 20 тесно посаженных SPAN-бисерин (`R_BERRY`) в форме капли (тир-дроп): острый угол в вершине у якорной ноды (обе ветви петли сходятся в одну точку — `teardropPerimeterPoints`, две касательные к окружности из внешней точки-вершины) и полукруглое закругление внизу (большая дуга той же окружности, `LOOP_CIRCLE_RADIUS = 62`, `LOOP_APEX_TO_CENTER = 140`). Бусины расставлены равномерно по длине контура (2 касательных отрезка + дуга), а не по параметру угла/дуги отдельно — иначе шаг был бы неровным на стыке прямых и дуги. Размер капли подобран так, чтобы шаг между соседними бисеринами по контуру (`totalLen / (count+1)`) был чуть больше их диаметра (`2·R_BERRY = 23`) — бусины соприкасаются вплотную, но не перекрывают друг друга. `LOOP_APEX_Y = 18` — координата самой вершины (невидимая точка схождения касательных), а не первой бусины: первая бусина лежит дальше по касательной (~19% её длины от вершины), поэтому итоговое расстояние от ноды до первой видимой бусины ≈40 единиц — то же, что и у dy=40 первой бусины Drop/Rhombus/Cluster. Первая (`i=0`) и последняя (`i=19`) бусины лежат ближе всего к вершине, по разные стороны от неё. `links` — последовательная цепочка `[i, i+1]` плюс замыкающая пара `[19, 0]` (обе ветви физически смыкаются у вершины рядом с якорем), поэтому вся петля — один связный компонент, подключённый к сетке через `beads[0]` (единственная бусина, которая по конвенции подвесок касается ноды-якоря).
*   **Размер подвески** ([src/data/pendantTemplates.ts](data/pendantTemplates.ts)): Шаблоны заданы в макетных координатах, где обычная бусина имеет радиус 18. При рендере всё умножается на `PENDANT_SCALE = BEAD_THEME.sizes.nodeRadius / 18`, чтобы нормальная бусина совпала с размером ноды сетки. Шаблоны поддерживают несколько размеров: `R = 18` (обычная), `R_BIG = 23.5` (крупная фокусная), `R_BERRY = 11.5` (мелкая грозди), `R_CLUSTER = 14` (бусина тройной грозди).
*   **Размещение подвесок:** Реестр `pendantPlacements` (`PendantPlacement[]`, список объектов `{ placementId, templateId, col, colorMap }`). Подвеска крепится к нижнему узлу `col`-й колонны последнего ряда (`r = 2·height`). На каждой колонне может быть только одна подвеска — новая заменяет старую. Цвета отдельных бусин в подвеске хранятся в `colorMap` (индекс в `template.beads` → цвет). Размещение переживает перезагрузку страницы (`localStorage`, ключ `silyanka:pendantPlacements`).
*   **Взаимоисключение с Bottom Chain:** см. «Взаимоисключение с подвесками» в разделе «Нижняя цепочка (Bottom Chain)» — подвески и Bottom Chain не могут быть включены одновременно.
*   **Управление подвесками:** Боковая панель (Sidebar) справа с кнопками для добавления шаблонов на конкретные колонны. Клик по шаблону в панели добавляет его на выбранную колонну (определяется через `hoveredCol` при наведении мыши). Удаление осуществляется нажатием на иконку в самом шаблоне на канвасе. Кнопка «×» справа от заголовка раздела «Pendants» очищает только подвески (`onClearAll`), не трогая Bottom Chain и декор-полосы; задизейблена при отсутствии размещений.
*   **Покраска бусин подвески:** При активном инструменте карандаш / ластик клик (или тап) на бусину подвески покрашивает её активным цветом или стирает (как обычные SPAN-бусины). Зажать и провести по нескольким бусинам — потоковое рисование, работает и мышью, и пальцем (см. «Ввод: мышь и тач» выше).
*   **Mirror Mode для подвесок:** В зеркальном режиме подвеска добавляется на обе симметричные колонны одновременно (логическое зеркало: `col c ↔ col (width-1-c)`). Покраска одной бусины также зеркалится. Удаление/перемещение одной подвески влияет на её парную (сохраняются как логическая пара для синхронизации).
*   **Трансформации при изменении сетки:** При уменьшении `width` подвески на исчезнувших колонках удаляются. В Mirror Mode при добавлении/удалении колонн все подвески сдвигаются вместе с рисунком (см. раздел 3 → *Зеркальный режим*).
*   **Боковая панель (PendantsSidebar):** Отрисовывается слева от хедера при нажатии кнопки с иконкой `Layers`. Содержит превью всех шаблонов, переключатель выбранной колонны (индикатор `hoveredCol`), кнопку удаления для каждого активного размещения и общую кнопку «Reset all». Каталог никогда не хранит цвет — превью бусин всегда рисуются в «пустом» стиле (см. ниже).
*   **Структура панели:** Над скроллящейся областью — закреплённый заголовок панели («Pendants & Decor»). Единая прокручиваемая панель без переключения вкладок, четыре визуально разделённых линиями-разделителями раздела сверху вниз — (1) тумблер Bottom Chain, (2) каталог подвесок (сетка 3 колонки, компактные карточки: превью ~52px, название до 2 строк), (3) цепочки-подвески (кнопка-тоггл «Pick chain nodes» + список «Placed»; см. «Цепочки-подвески (Pendant Chains)» ниже), (4) декор-полосы (карточка Band + список «Placed» с ±). Каждый раздел имеет собственный заголовок и короткое англоязычное описание (`pendants-sidebar__section-title` / `-desc`): «Bottom Chain» — Decorative edge added below the last row; «Pendants» — Drag a template onto a bottom-row node; «Chains» — Drape a chain between two bottom-row nodes (текст описания меняется на подсказку по шагам выбора, пока инструмент активен); «Decor» — Drag a band onto a gap between rows. Все четыре раздела скроллятся как единая область (`pendants-sidebar__body`); футер с кнопкой «Reset all» и контекстной подсказкой закреплён снаружи скролла и не прокручивается вместе с содержимым.
*   **Точечная очистка разделов (`pendants-sidebar__section-clear`):** Справа от заголовка каждого из четырёх разделов (`pendants-sidebar__section-heading-row`, `justify-content: space-between`) — компактная кнопка «×», очищающая **только** этот раздел, в отличие от общей «Reset all» в футере (которая сбрасывает все состояния разом): «Bottom Chain» → вызывает `onBottomEdgeToggle` (задизейблена при `!bottomEdgeEnabled` — нечего выключать); «Pendants» → `onClearAll` (задизейблена при отсутствии размещений); «Chains» → `onClearChains` (задизейблена при отсутствии цепочек); «Decor» → `onClearDecor` (задизейблена при отсутствии активных полос). Подсказка про включение Bottom Chain при наличии подвесок теперь ссылается на точечную очистку раздела «Pendants», а не на общий «Reset all» (очистка одних только подвесок не обязана сбрасывать декор).
*   **Визуальный стиль бусин подвески:** Бусины подвески переиспользуют те же CSS-классы (`bead`, `bead--type-node`/`bead--type-span`, `bead--empty`, `bead__body`) и переменную `--bead-color`, что и обычная сетка (`BeadView`) — распределены по обёртке `<g>` (тип/пустота) и самому телу (`bead__body`), а не на одном элементе, иначе CSS-селекторы вида `.bead--empty .bead__body` (селектор потомка) не сработают. Благодаря этому непокрашенная подвеска выглядит как полупрозрачный контур (как пустой узел сетки) вместо невидимого `transparent`-круга, а покрашенный NODE получает то же свечение (`drop-shadow`). Каталог шаблонов в сайдбаре (`PendantPreview`, `BandPreview`) всегда рисуется в состоянии `bead--empty`, т.к. это шаблоны, а не размещения.

#### *Цепочки-подвески (Pendant Chains)*
Отдельный от точечных подвесок вид декора: провисающая полукругом дуга SPAN-бисерин между **двумя** произвольными узлами нижнего ряда (не обязательно соседними), а не одна подвеска на одном якоре. Генератора не касается — как и точечные подвески, рендерится отдельным слоем поверх готовой сетки.
*   **Модель** ([src/types/pendant.ts](types/pendant.ts)): `PendantChain { placementId, startCol, endCol, colorMap }` — `startCol < endCol` (нормализуется при создании). От одной ноды может отходить сколько угодно цепочек; цепочка и точечная подвеска могут сосуществовать на одной ноде — никакого взаимного исключения между ними нет (в отличие от Bottom Chain vs Pendants). Реестр `pendantChains` — `usePersistedState('silyanka:pendantChains', ...)` в [useSilyankaProject.ts](hooks/useSilyankaProject.ts), управляется хуком [usePendantChains.ts](hooks/usePendantChains.ts) (по образцу `usePendants.ts`): `addChain`, `removeChain`, `paintChainBead`, `clearAllChains`.
*   **Геометрия** ([src/utils/pendantChain.ts](utils/pendantChain.ts)):
    *   **Число бисерин** зависит от расстояния между узлами-креплениями и не задаётся пользователем напрямую: `getChainBeadCount(distance) = max(1, round(distance / pitch) - 1)`, где `pitch = spanRadius·2 + 2` (тот же принцип, что даёт `minBeadPitch` в `generator.ts`) — далёкие друг от друга узлы дают более длинную цепочку.
    *   **Провис**: `computeChainBeadPositions` — тот же sin-профиль, что и `generateArcSpan` (`y += sagDepth · sin(π·t)`), но глубина `sagDepth = sagScale · distance^sagExponent` — **нелинейно** зависит от расстояния между узлами (`sagScale = 3.5`, `sagExponent = 0.5`, [theme.ts](config/theme.ts) → `pendantChainDefaults`). При `sagExponent < 1` отношение `sagDepth/distance` убывает с ростом расстояния: цепочка между 3 нодами может провисать глубоко относительно своей длины, а между 50 нодами — уже не так глубоко (линейный рост `sagDepth = k·distance` дал бы неестественно растянутую дугу на большом расстоянии). В отличие от тонкой фиксированной кромки Top/Bottom Edge, это самостоятельная дуга произвольного размаха.
*   **Рендер** ([PendantChainLayer.tsx](components/Editor/PendantChainLayer/PendantChainLayer.tsx)): бисерины типа SPAN, красятся/стираются как обычные пролёты (переиспользуют классы `bead`, `bead--type-span`, `bead--empty`, `bead__body`). Кнопка удаления — под самой нижней (провисшей) бисериной цепочки, видна по hover на `.pendant-chain-group` (только по наведению на сами бисерины/кнопку — без вспомогательного invisible-rect, в отличие от `PendantLayer`: такой прямоугольник неизбежно перекрывал бы саму нижнюю строку узлов, по которой ведётся покраска и выбор нод новой цепочки, блокируя клики).
*   **Инструмент выбора узлов:** кнопка «Pick chain nodes» в разделе «Chains» сайдбара (`PendantsSidebar`) переключает `activeTool` в `'pendant-chain'` (добавлен в `DrawingTool`, [useDrawing.ts](hooks/useDrawing.ts)). В этом режиме обычная покраска на канвасе отключена; клик по узлу нижнего ряда отмечает начало (подсвечивается как обычный highlighted-узел), клик по другому узлу нижнего ряда — конец, после чего цепочка создаётся и выбор сбрасывается (тул остаётся активным — можно сразу продолжать ставить следующие цепочки). Повторный клик по уже отмеченному узлу отменяет незавершённый выбор. `Escape` и переключение на любой другой инструмент тоже сбрасывают незавершённый выбор (`chainPendingStart`, `useSilyankaProject.ts`) — та же схема, что и у сброса штампа.
*   **Заливка (Flood Fill):** цепочки подключены к унифицированному графу заливки ([floodFill.ts](utils/floodFill.ts)) наравне с сеткой и точечными подвесками — каждая цепочка образует линейную цепь `startNode → bead-0 → … → bead-(N-1) → endNode`, где оба конца — обычные узлы нижнего ряда; цвет может «перетекать» из сетки в цепочку и обратно через любой из её концов.
*   **Undo/Redo:** `pendantChains` — третье поле `HistorySnapshot` (наравне с `designMap` и `pendants`) в `useDrawing.ts` — один Undo/Redo откатывает изменения сетки, подвесок и цепочек синхронно. Создание/удаление самой цепочки (двухклик на канвасе, крестик на цепочке) в историю **не** пишется — так же, как и у точечных подвесок; отменяется через Undo только покраска бисерин цепочки.
*   **Mirror Mode:** нижний ряд чётный — зеркало точное (`col c ↔ col (width-1-c)`), поэтому при создании цепочки в зеркальном режиме сразу добавляется и её зеркальная пара `(width-1-endCol, width-1-startCol)`. Покраска, удаление и заливка одной цепочки зеркалят вторую.
*   **Изменение сетки:** при сужении `width` цепочка удаляется целиком, если хотя бы один из её концов вышел за границу (в отличие от точечных подвесок, у цепочки нет частичного состояния). В Mirror Mode при добавлении/удалении колонн оба конца сдвигаются вместе с рисунком.
*   **Replace Color:** цепочки участвуют в замене цвета по клику на плашку статистики (`swapColorInChains`, [utils/colorSwap.ts](utils/colorSwap.ts)) — та же механика, что и `swapColorInPendants`.


#### *Hitbox-адаптация*
Для удобства работы с мелкими элементами активная зона клика (`hitboxRadius = 11px`) значительно превышает визуальный размер бисерины (`nodeRadius = spanRadius = 6px` — оба типа отрисовываются с одинаковым радиусом, визуального различия по размеру между NODE и SPAN нет).

#### *Зеркальный режим (Mirror Mode)*
Режим симметричного рисования, включается кнопкой в хедере (иконка `FlipHorizontal`). При активации на канвасе отрисовывается вертикальная пунктирная ось симметрии по центру (`x = maxX / 2`, где `maxX` — максимальная координата узлов чётного ряда). Любой клик или мазок по бисерине автоматически закрашивает и её зеркальную пару — оба изменения попадают в Design Map за **один** шаг истории (один Undo отменяет пару целиком). Работает и с ластиком: стирание зеркалит так же. Покраска бусин подвески тоже зеркалится.

Логика зеркала различается для двух типов бисерин ([src/utils/mirror.ts](utils/mirror.ts)):
*   **NODE — логическое зеркало:** 
    *   Чётные ряды (не смещены): `col c ↔ col (width-1-c)`.
    *   Нечётные ряды (смещены на `stepX/2`): `col c ↔ col (width-2-c)`, диапазон `c = -1..width-1` (нечётный ряд шире чётного на один узел с каждого края, см. раздел 2). Крайний узел `col = width-1` зеркалится в `col = -1` (и наоборот) — у каждого узла нечётного ряда есть пара.
    Из-за того, что нечётные ряды смещены на `stepX/2`, зеркальные пары в нечётных рядах визуально не равноудалены от оси — это реальная геометрия плетения силянки.
*   **SPAN — геометрическое зеркало:** 
    *   **Верхняя кромка** (r=0): индексы бисерин внутри `internalTop + 1` переворачиваются симметрично: `bead_i ↔ bead_(internalTop+1-i)`.
    *   **Диагональные грани ромба** (рёбра между рядами): пролёты отражаются по фактической геометрии с обменом сторон `left ↔ right`.
    *   **Декоративные полосы** (`decor-r-k-c`): повторяют разметку узлового ряда `r`, поэтому логическое зеркало как у NODE.
    Логическое зеркало для пролётов невозможно — оно порождало бы грани, которых нет в графе силянки.

Состояние режима сохраняется в `localStorage` (ключ `silyanka:mirrorMode`). При `width ≤ 1` ось не отрисовывается и зеркалирование не выполняется.

**Изменение ширины в Mirror Mode:** при включённом режиме кнопки Width ± меняют ширину на **±2** (добавляют/убирают по колонке с каждой стороны), чтобы рисунок остался по центру относительно новой оси ([src/App.tsx](App.tsx): `updateDimension` → `shiftDesignMapColumns`). Design Map автоматически пересчитывается функцией [src/utils/regrid.ts](utils/regrid.ts): `shiftDesignMapColumns(designMap, delta, newW)` сдвигает все ID бисерин на соответствующее число колонок; бисерины, вышедшие за границу сетки, отбрасываются. Все подвески также сдвигаются вместе с рисунком (`col += delta`). Пересчёт записывается в историю и отменяется через Undo.


#### *Zoom System*
Масштабирование холста от 25% до 300% с шагом 10%. Реализовано **не CSS-трансформацией**, а прямым умножением атрибутов `width`/`height` корневого `<svg>` на `zoom` при неизменном `viewBox` ([CanvasView.tsx](components/Editor/CanvasView/CanvasView.tsx): `width={dim.w * zoom}`, `height={dim.h * zoom}`) — поэтому SVG-границы остаются векторно-чёткими на любом уровне зума. Из-за этого зум завязан на scroll-позицию контейнера `.canvas__svg`, а не на CSS translate/scale. В [src/utils/zoomToPoint.ts](utils/zoomToPoint.ts) есть утилита `computeZoomToPointScroll` для пересчёта scroll-координат при zoom-to-point, но в текущем pinch-жесте она не используется — там своя, более простая формула через дельту midpoint (см. ниже).

**Двупальцевый pinch-жест (тач)** — [src/hooks/useTouchPanZoom.ts](hooks/useTouchPanZoom.ts), общий хук для `CanvasView` и `CrossWeaveCanvasView`. Во время самого жеста `zoom` **не идёт через React state**: `width`/`height` `<svg>` и `scrollLeft`/`scrollTop` контейнера пишутся напрямую в DOM на каждый `pointermove`, а в React state (проп `zoom`, персистентный через `usePersistedState` → `localStorage`) новое значение коммитится **один раз**, когда жест заканчивается (счётчик активных пальцев падает ниже 2, через колбэк `onSetZoom`/`setZoomAbsolute`). Причина — тот же паттерн, что и у драга/ресайза `ReferenceWindow` (см. ниже): один физический пинч даёт минимум два `pointermove` в кадр (по одному на палец), и если бы каждый коммитил React state, это перерисовывало бы всё дерево от `App` (включая сотни `BeadView`) и синхронно писало в `localStorage` на каждый кадр — источник подвисаний холста на мобилке. Live-масштаб клампится к `APP_CONSTRAINTS.min/maxZoom` уже во время жеста (не только на коммите), чтобы не было визуального «скачка» в конце. Wheel-zoom (`useWheelZoom`, `Ctrl`+колесо) на этот паттерн не переведён — коммитит `zoom` через `onZoomChange` на каждое событие, как и раньше.

#### *Тема холста (Dark / Light)*
Переключатель тёмной/светлой темы **только для холста** — карточки `.canvas__svg` и её содержимого (бисер, линейки, ось зеркала). Хедер, сайдбар и внешняя рама (`.editor__viewport`) всегда тёмные: светлый холст читается как «лист бумаги» на тёмной раме.
*   **Состояние:** `canvasTheme: 'dark' | 'light'` в [src/App.tsx](App.tsx), персистентно в `localStorage` (ключ `app:canvasTheme`). Дефолт при чистом хранилище — `dark` (системная тема не учитывается). Переключается плавающей кнопкой ☀/🌙 **у холста** — в правом нижнем углу над «Download PNG» (`.canvas-theme-toggle` в `CanvasView`), не в хедере.
*   **Механизм:** `data-canvas-theme={canvasTheme}` вешается в [CanvasView](components/Editor/CanvasView/CanvasView.tsx) **на два элемента**: на карточку `<div className="canvas__svg">` (отсюда токены каскадятся вниз ко всем бусинам/линейкам) и на раму `<main className="editor__viewport">` (ancestor карточки — CSS не умеет выбирать предка по потомку, поэтому фон рамы задаётся своим атрибутом). Тема реализована **через CSS-переменные**: тёмные значения — дефолт (в `.canvas__svg` и в `var(...)`-fallback у бисера/оси/чипов), светлые — переопределяются селектором `[data-canvas-theme="light"]`.
*   **Светлая палитра:** поверхность холста `#F5F5F7` (нейтральный светло-серый), граница/скроллбар — приглушённый slate. **Рама (`.editor__viewport`)** остаётся тёмной, но менее чёрной — navy `#0e1b33` вместо `#020617`, чтобы снизить контраст со светлой карточкой. Базовый (незакрашенный) бисер, чей видимый вид задаётся правилами `.bead--empty` в [BeadView.css](components/Editor/BeadView/BeadView.css) (в модели `defaultColorFor` = `transparent`), рисуется **чистым контуром без заливки и без свечения** (мутная полупрозрачная заливка и серый glow на светлом фоне выглядят «грязно»): span — мягкий slate-контур, нода — более тёмный и плотный. **Закрашенная нода** в тёмной теме подсвечивается glow собственным цветом (`drop-shadow(0 0 2px var(--bead-color))`) — «pop» ярких бусин; в светлой теме этот glow **приглушается** (радиус размытия 1px вместо 2px), иначе тёмные бусины на светлом фоне дают мутный ореол — «pop» ноды сохраняется, но мягче. Экспорт PNG повторяет это правило через поле `nodeGlow` в палитрах [exportScheme.ts](utils/exportScheme.ts). Ось зеркала (белый пунктир, невидимый на светлом) переопределяется в тёмный (`--canvas-mirror-axis`). Чипы span-контролов на белом осветляются (розовый/голубой tint вместо тяжёлых тёмных заливок).

#### *Reference Window*
Плавающее окно с картинкой-образцом поверх холста — для сверки узора при плетении. Технико-независимо (доступно и в silyanka, и в crossWeave), переживает перезагрузку страницы.
*   **Состояние:** `open` — единственное поле, поднятое в `App.tsx` (`usePersistedState`, ключ `app:referenceWindow:open`), т.к. нужно хедеру для подсветки кнопки-тоггла. Всё остальное — [src/hooks/useReferenceImage.ts](hooks/useReferenceImage.ts) (`position {x,y}`, `size {w,h}`, `zoom`, `collapsed`, персистентны в `localStorage`, ключи `app:referenceWindow:position|size|zoom|collapsed`) — подключается **внутри самого `ReferenceWindow.tsx`**, а не в `App`: это состояние меняется на каждый pointermove/wheel/resize, и если бы оно жило в `App`, каждое такое событие перерисовывало бы всё дерево (включая холст с сотнями бисерин) → подвисания при перетаскивании/зуме/ресайзе окна. Сама картинка в `localStorage` не попадает.
*   **Хранение картинки:** [src/utils/referenceImageDb.ts](utils/referenceImageDb.ts) — нативная обёртка над IndexedDB (без npm-зависимостей), одна запись (`getImage`/`putImage`/`deleteImage`). При загрузке файл даунскейлится через `<canvas>` (`createImageBitmap` → длинная сторона ≈2000px → `toBlob('image/jpeg', 0.85)`) перед записью — оригиналы со смартфонов не раздувают базу. Если запись в IndexedDB не удалась (переполнена квота, приватный режим) — картинка остаётся видимой только в памяти текущей сессии (`ObjectURL`) и пропадает при перезагрузке; в шапке окна показывается предупреждение (`hasPersistError`).
*   **Компонент:** [src/components/Editor/ReferenceWindow/ReferenceWindow.tsx](components/Editor/ReferenceWindow/ReferenceWindow.tsx). `position: fixed`, дефолтная позиция при первом открытии — под хедером, правее холста. Динамические `position`/`size`/`zoom` передаются через CSS-переменные (`--ref-x/y/w/h`, `--ref-zoom`).
    *   **Драг:** шапка окна — `onPointerDown` + `setPointerCapture` + `onPointerMove`, позиция клампится в границы вьюпорта. Во время движения координаты пишутся напрямую в DOM (`style.setProperty('--ref-x'/'--ref-y', ...)`, мимо React state) — в `usePersistedState` (и `localStorage`) коммитится только финальное значение на `pointerup`.
    *   **Ресайз:** собственный `.reference-window__resize-handle` (уголок внизу справа) поверх Pointer Events — тот же приём "пишем в DOM во время драга (`--ref-w`/`--ref-h`), коммитим в state на `pointerup`", что и у драга шапки. Раньше был нативный CSS `resize: both` — он управляется только мышью и не откликается на тач ни на Android, ни на iOS, из-за чего окно нельзя было уменьшить с телефона; заменён на собственный хэндл, одинаково работающий для мыши/тача/пера. `ResizeObserver` на корневом элементе остаётся как подстраховка (синхронизирует размер в состояние с debounce 200мс), но при свёрнутом окне (см. ниже) отключается — иначе схлопнутая до шапки высота записалась бы как `size`.
    *   **Сворачивание:** кнопка `ChevronDown`/`ChevronUp` в шапке (между зум-контролами и крестиком закрытия) сворачивает окно до одной шапки — тело картинки, футер и resize-хэндл не рендерятся, высота становится `auto` вместо зафиксированной `--ref-h`. Повторный клик разворачивает обратно к прежнему размеру (`size` не трогается сворачиванием). Состояние `collapsed` персистентно (`app:referenceWindow:collapsed`), окно можно продолжать перетаскивать за шапку в свёрнутом виде.
    *   **Зум картинки:** независим от зума холста (25%–400%, шаг 10%) — `Ctrl`+колесо (`useWheelZoom`, переиспользован из `CanvasView`) поверх области картинки и кнопки ± в шапке; скрыт вместе с телом окна при сворачивании.
    *   **Панорамирование:** прокрутка контейнера картинки (`overflow: auto`), картинка масштабируется через `transform: scale()`.
    *   **Пустое состояние:** dropzone (drag&drop файла) + кнопка выбора файла. **Заполненное состояние:** кнопки «Заменить»/«Убрать».
*   **Кнопка-тоггл:** в хедере, технико-независимая секция (иконка `Image` из `lucide-react`), в конце хедера — рядом с кнопкой подвесок (`PendantIcon`), после блока Undo/Redo/Clear.

#### *Сохранение/загрузка проекта*
Экспорт/импорт всего проекта одним файлом — [src/utils/projectFile.ts](utils/projectFile.ts). Не заводит собственное хранилище состояния: всё состояние приложения уже лежит в `localStorage` под префиксами `app:`/`silyanka:`/`crossWeave:` (см. `usePersistedState` во всех разделах выше), файл проекта просто упаковывает и восстанавливает эти ключи.
*   **Формат файла** (`silyanka-project.json`): `{ version: 1, savedAt: ISO-строка, localStorage: Record<ключ, значение> }`. Картинка Reference Window (хранится отдельно в IndexedDB, см. выше) в файл **не включается** — при загрузке проекта окно референса восстановит позицию/размер/зум, но не саму картинку.
*   **Сборка данных (`buildProjectData`):** Собирает все ключи `localStorage`, начинающиеся с `app:`/`silyanka:`/`crossWeave:`, в объект `ProjectFile`. Переиспользуется и файловым экспортом, и Share-ссылкой (см. ниже).
*   **Сохранение (`exportProject`):** `buildProjectData()` → JSON → скачивание как файл (`Blob` + временная ссылка `<a download>`, тот же паттерн, что и у экспорта PNG — см. раздел 4).
*   **Применение данных (`applyProjectData`):** Сначала удаляет **все** существующие ключи `app:`/`silyanka:`/`crossWeave:` (не только те, что есть в новых данных — иначе от предыдущего проекта могли бы остаться «хвосты»), затем записывает новые пары. Общая часть для файловой загрузки и Share-ссылки.
*   **Загрузка (`importProject`):** Парсит и валидирует файл (проверка `version` и формы `localStorage`); при ошибке кидает исключение с текстом для пользователя, `localStorage` не трогается. При успехе вызывает `applyProjectData`, затем страница перезагружается (`location.reload()`), и всё состояние подхватывается штатными проверками в `usePersistedState`.
*   **UI:** Кнопки-иконки `Download`/`Upload` (`lucide-react`) в хедере, в общем блоке с Undo/Redo/CLEAR. Load открывает системный выбор файла (скрытый `<input type="file" accept="application/json">`); после выбора файла — `window.confirm('Current work will be replaced, continue?')`, и только при подтверждении вызывается `importProject`.

#### *Share-ссылка*
Альтернатива файлу — ссылка с зашитой схемой, без необходимости пересылать файл. Логика в [src/utils/shareLink.ts](utils/shareLink.ts).
*   **Формат данных:** тот же `ProjectFile`, что и в файле проекта (без картинки референса — она хранится отдельно в IndexedDB и слишком тяжёлая для URL), сериализован в JSON, сжат `CompressionStream('gzip')` (нативный Compression Streams API, без новых зависимостей на клиенте) и закодирован в base64url.
*   **Короткая ссылка через бэкенд (`api/share.ts`):** сжатый payload отправляется `POST`-ом на свой Vercel Edge Function, которая кладёт его в Redis (**Upstash Redis** через Vercel Marketplace-интеграцию, пакет `@upstash/redis`, `Redis.fromEnv()`) под случайным 7-символьным id (`[a-zA-Z0-9]`, коллизии проверяются `exists` и перегенерируются) и возвращает этот id. Итоговая ссылка — `.../#g=<id>` (7 символов вместо десятков КБ). Записи в Redis бессрочные (без TTL) — короткая ссылка не истекает. Тот же контракт продублирован в [server.ts](../server.ts) (`app.post('/api/share')` / `app.get('/api/share')`) для локальной разработки через `npm run dev:server` (Vite-прокси `/api` → `localhost:3001`, см. [vite.config.js](../vite.config.js)) — на самом Vercel это Edge Function, локально — обычный Express-роут, оба используют один и тот же Redis через переменные окружения (`UPSTASH_REDIS_REST_URL`/`UPSTASH_REDIS_REST_TOKEN`, либо legacy `KV_REST_API_URL`/`KV_REST_API_TOKEN` — `Redis.fromEnv()` подхватывает любые).
*   **Откат на длинную ссылку:** при любой сетевой ошибке или недоступности бэкенда (`saveToBackend` вернул `null`) — молчаливый откат на длинную ссылку с данными прямо в `#`-фрагменте (`.../#s=<данные>`, без похода на сервер вообще). Раньше вместо своего бэкенда использовалось сокращение через публичный is.gd (JSONP) — отказались из-за жёсткого лимита в 5000 символов на URL: для крупных схем итоговый запрос к `create.php` где-то по пути (прокси/CDN) возвращал HTML-страницу ошибки (414) вместо JSON, а наш JSONP `<script>` наивно пытался исполнить её как JS и падал с `SyntaxError`. Свой бэкенд такого лимита не имеет (Redis-значение проверяется только на разумный верхний предел `MAX_PAYLOAD_LENGTH = 200_000` символов — с большим запасом).
*   **Копирование в буфер и обратная связь:** `navigator.clipboard.writeText` обёрнут в `try/catch` — сетевой запрос к бэкенду перед копированием может занять заметное время, из-за чего браузер иногда успевает снять «user activation», выданную кликом, и запись в буфер молча отклоняется (`NotAllowedError`, характерно для боевого домена). При успехе показывается маленький тост-уведомление «Link copied» ([src/components/Toast/Toast.tsx](components/Toast/Toast.tsx), автоскрытие через 2.2 сек) — независимо от того, короткая ссылка получилась или откат на длинную. При отказе клипборда — `window.prompt()` с готовой ссылкой, чтобы пользователь мог скопировать её вручную вместо молчаливого провала шеринга.
*   **Загрузка по ссылке:** при монтировании `App` разбирает `location.hash` (`parseShareHash`) — если хэш начинается с `#g=`, id запрашивается у бэкенда (`GET /api/share?id=...`, 404 → ссылка не найдена/устарела); если с `#s=` — данные уже лежат в самом хэше. Хэш сразу очищается через `history.replaceState` (чтобы `confirm` не всплывал повторно при F5/навигации), затем `window.confirm('Load pattern from link? Current work will be replaced.')`; при подтверждении — `applyProjectData` + `location.reload()`.
*   **UI:** кнопка-иконка `Share2` (`lucide-react`) в хедере рядом с Download/Upload.
*   **Ограничения:** если браузер не поддерживает `CompressionStream`/`DecompressionStream` (очень старые версии) — создание ссылки кидает читаемую ошибку, полифилл сознательно не добавлен. Работа бэкенда зависит от наличия провизионированного Upstash Redis в Vercel-проекте (Marketplace-интеграция) — без неё `saveToBackend` всегда вернёт `null`, и шеринг тихо откатится на длинную ссылку (не баг, штатное вырождение).

### **История действий (Undo/Redo)**
Система записывает историю на уровне мазков, а не отдельных кликов. Стек живёт в [useDrawing.ts](hooks/useDrawing.ts) и покрывает **и** Design Map сетки, **и** размещения подвесок (`pendantPlacements`), **и** цепочки-подвески (`pendantChains`, см. также «Цепочки-подвески» в разделе 3) — один снимок истории (`HistorySnapshot`) хранит связку `{ designMap, pendants, chains }`, поэтому один Undo/Redo синхронно откатывает изменения всех трёх типов, сделанные за один жест (например, мазок карандашом, задевший и сетку, и подвеску, или заливка, окрасившая оба сразу):
*   **Запись в историю:** Состояние `{ designMap, pendants, chains }` фиксируется в момент начала мазка (`startDrawing`, вызывается при `mousedown` на канвасе — до разделения на «клик по сетке» / «клик по подвеске»/«клик по цепочке»). При завершении мазка (`stopDrawing`, `mouseup`/`mouseleave`), если изменилась карта **или** размещения подвесок **или** цепочки, сохранённый снимок добавляется в стек `past`. Операция `clearAll` тоже добавляет снимок в историю перед очисткой, но только если было что чистить (карта, хотя бы одна подвеска с непустым `colorMap`, или хотя бы одна цепочка с непустым `colorMap`) — холостого снимка не возникает.
*   **Заливка (Flood Fill):** Использует общий метод `applyPatch(designMapFn, pendantsFn)`, который применяет обе функции-патчера (любая может быть `null`, если тип не задет) и пишет **один** снимок в историю, даже если заливка одновременно перекрасила бусины сетки и подвески (через якорную ноду).
*   **Стек истории:** Хранит до 30 снимков (`MAX_HISTORY = 30`). При переполнении самый старый снимок удаляется.
*   **Redo-стек:** Стек `future` сбрасывается при каждом новом мазке или очистке. Undo перемещает текущее состояние (обе части) в `future`, Redo — возвращает обратно.
*   **Управление:** Кнопки ↩ / ↪ в хедере (с блокировкой при пустых стеках). Клавиатурные сокращения работают как с `Ctrl`, так и с `Cmd` (macOS): `Ctrl/Cmd+Z` — отменить, `Ctrl/Cmd+Y` или `Ctrl/Cmd+Shift+Z` — повторить.
*   **Вне истории:** Структурные изменения — добавление/удаление подвески (`addPlacement`/`removePlacement`/`clearAllPlacements` в `usePendants`), изменение Width/Height/Top/Bottom Edge, декор-полос — в стек не попадают и Undo не отменяются (кроме сдвига `designMap` при Width ±2 в Mirror Mode, который идёт через `remapDesignMap` и пишется в историю; сопутствующий сдвиг колонок подвесок при этом в историю не попадает).

### **Интерфейс управления (Header)**
Фиксированная панель инструментов в верхней части экрана ([src/components/Editor/Header/Header.tsx](components/Editor/Header/Header.tsx)):
*   **Палитра цветов:** 5 хардкод-цветов (`#ff4757`, `#ffd32a`, `#22d3ee`, `#e879f9`, `#ffffff`). Клик по любому цвету активирует карандаш и устанавливает этот цвет как активный. Клики по базовой палитре **не** записываются в Recent Colors.
*   **Recent Colors:** 5 слотов второй строкой под базовой палитрой. Накапливают цвета, выбранные через Custom Picker / EyeDropper, с дедупликацией, FIFO-вытеснением и persistance в `localStorage` (см. раздел 3 → *Recent Colors*).
*   **Custom color picker:** Кнопка с конической градиентной заливкой (`conic-gradient(from 0deg, #ff4757, #ff9f43, #ffd32a, #2ed573, #22d3ee, #1e90ff, #e879f9, #ff4757)`) — открывает поповер `ColorPicker.tsx` с двумя вкладками:
    *   **Pick:** SV-квадрат + Hue-полоса + hex-инпут (ручной подбор одного цвета). При подтверждении (`Confirm`, `Enter`) цвет добавляется в Recent Colors; инструмент переключается на карандаш только если до этого был активен ластик — заливка/штамп остаются активными. Если активный цвет находится вне базовой палитры, кнопка показывает этот цвет вместо градиента (индикатор «работаем кастомным цветом»).
    *   **Generate:** ИИ-генерация палитры через Colormind API (`src/utils/colormindApi.ts`, запрос идёт на относительный путь `/api/generate-palette` — в проде это Vercel-функция [api/generate-palette.ts](../api/generate-palette.ts), локально Vite проксирует этот путь на `server.ts`) — кнопки 2/3/5 цветов, сгенерированные цвета — квадратные свотчи, кликом можно исключить/включить каждый обратно в выбор (состояние выбора — рамка + тень, без иконки поверх цвета). После первой генерации появляется кнопка **Regenerate** (иконка обновления) — перегенерирует палитру тем же количеством цветов без повторного клика по 2/3/5. Под каждым свотчем — ряд из двух всегда видимых мини-кнопок: замо́к (лочит/анлочит цвет: закреплённые слоты Colormind возвращает без изменений и подбирает остальные цвета под них при следующей генерации/Regenerate) и пипетка, открывающая компактный поповер со своим SV-квадратом/Hue/hex (уменьшенная копия вкладки Pick) и явной кнопкой **Apply** — применяет цвет, закрепляет его (лочит) и помечает выбранным; закрывается по Apply, клику вне поповера или Escape (независимо от всего ColorPicker, не закрывая его). На ≤767.98px оба поповера (основной `ColorPicker` и мини-пикер вставки цвета) анкерятся по `position: fixed` к верху viewport (под шапкой, `top: calc(var(--palette-height) + 12px)`, `max-height` до низа экрана с `overflow-y: auto`), а не по анкеру к своей кнопке-триггеру — на узких экранах эта кнопка часто стоит у самого края экрана (особенно внутри мобильного попапа палитры, см. ниже), и центровка-под-кнопкой выталкивала 220-260px попап за границу экрана. Якорь именно к верху, а не к вертикальному центру экрана — центр viewport на телефоне приходится на середину холста, и попап там выглядел как случайно вырезанная "дыра" посреди канваса, а не как модалка, растущая из кнопки в шапке. На этом же брейкпоинте под основным `ColorPicker` рендерится затемняющая подложка (`.color-picker-backdrop`, `Header.tsx`); клик по подложке закрывает пикер. Кнопка **Replace Palette** **заменяет** всю основную палитру (`palette` в `App.tsx`, персистится как `app:palette`) выбранными цветами — не добавляет к существующим. Если выбрано меньше 5 цветов (генерация «2» или «3»), оставшиеся до 5 слотов добиваются базовыми цветами (`#ffffff`, `#ff4757`, `#000000`), чтобы палитра не была слишком скудной для рисования.
*   **EyeDropper:** Кнопка-пипетка, появляется условно (только при наличии `'EyeDropper' in window`). Вызывает нативный селектор пикселя; выбранный цвет добавляется в Recent Colors; инструмент переключается на карандаш только если до этого был активен ластик. Отмена обрабатывается беззвучно. Ограничение платформы: `EyeDropper` API реализован только в десктопных Chromium-браузерах — ни Chrome для Android, ни Safari iOS его не поддерживают, поэтому на мобильных устройствах кнопка корректно не рендерится вовсе (не баг).
*   **Инструмент карандаш:** Кнопка `Pencil` — первая в панели инструментов. Клик всегда переводит `activeTool` в `'pencil'`; подсвечивается активной, когда карандаш выбран. Шорткат `B`.
*   **Инструмент ластик:** Кнопка ⌫, переключение между режимами карандаша и ластика. Поскольку ластик не использует цвет, выбор любого цвета (палитра, custom или recent) выводит из ластика обратно в карандаш. Если же был активен инструмент, использующий цвет (заливка, штамп), выбор цвета его не сбрасывает — меняется только активный цвет. Шорткат `E` (тоггл, как и клик).
*   **Reference Window:** Кнопка-тоггл (иконка `Image`) в конце хедера рядом с кнопкой подвесок, технико-независимая — открывает/закрывает плавающее окно с картинкой-образцом (см. раздел 3 → *Reference Window*).
*   **Инструмент заливка:** Кнопка `PaintBucket` между ластиком и штампом. Повторный клик возвращает в режим карандаша (см. раздел 3 → *Заливка (Flood Fill)*). Шорткат `G` (только silyanka; тоггл).
*   **Инструмент штамп:** Кнопка `Stamp` между заливкой и Mirror Mode. Повторный клик (или выбор любого другого инструмента) возвращает в режим карандаша и сбрасывает захваченный узор (см. раздел 3 → *Штамп (Stamp)*). Пока узор загружен и штамп активен, на углу самой кнопки `Stamp` появляется маленькая кнопка-бейдж переключения точки привязки (`ArrowUpToLine`/`ArrowDownToLine`, top/bottom) — намеренно не отдельная кнопка в общем ряду, а визуально дочерний элемент `Stamp` (`.tool-btn-group`). Шорткат `S` (только silyanka; тоггл — Shift для точки привязки штампа не конфликтует, это отдельная клавиша).
*   **Mirror Mode:** Кнопка-переключатель `FlipHorizontal` для включения/выключения симметричного рисования (см. раздел 3 → *Зеркальный режим*). Расположена сразу после инструмента штампа. При включённом режиме выводит статус в aria-pressed. Шорткат `M` (тоггл, работает в обеих техниках).
*   **Клавиатурные шорткаты инструментов:** Однобуквенные, Photoshop-style — `B` Pencil, `E` Eraser, `G` Flood Fill, `S` Stamp, `M` Mirror Mode (обрабатываются в `App.tsx` вместе с Undo/Redo/Escape/Alt/Shift). Не срабатывают при зажатых `Ctrl/Cmd/Alt`, при авто-повторе (`e.repeat`, зажатая клавиша) и когда фокус находится в поле ввода (`INPUT`/`TEXTAREA`/`contentEditable` — например, при редактировании числового значения Stepper'а). Смена инструмента через шорткат в silyanka идёт через ту же обёртку `setSilyankaTool`, что и клик по кнопке, — сбрасывает захваченный штамп при уходе с него.
*   **Контролы сетки (основная полоса):** Группы ± кнопок для изменения:
    *   **Width** (количество колонн, минимум 1, без верхнего предела): при Mirror Mode изменяет на ±2 (по колонке с каждой стороны). При отключённом режиме добавляет справа. Прямой ввод числа в поле (Stepper `onSet` → `setWidthAbsolute`) при Mirror Mode тоже симметричен: нечётная разница с текущей шириной округляется до чётной, чтобы сохранить центровку рисунка.
    *   **Height** (количество рядов, минимум 1, без верхнего предела): добавляет/удаляет рядов снизу.
    *   **Top Edge** (3–10) и **Bottom Edge** (3–10): глобальные параметры пролётов. Каждый имеет кнопку Reset (↻) для возврата к дефолту и очистки соответствующих per-row override'ов.
    Изменения пересчитываются мгновенно. При уменьшении высоты удаляются подвески и декор-полосы на исчезнувших рядах.
*   **Overflow меню** (при узком экране, ⋯): кнопка `MoreHorizontal` раскрывает панель с Width/Height/Top Edge/Bottom Edge (со своими Reset кнопками) — видна с ≤1279.98px. Панель общая для обеих техник (раньше рендерилась только для silyanka — CrossWeave на ≤1279.98px оставался без доступа к Width/Height, это заодно исправлено).

На ≤767.98px этого недостаточно (сумма блоков ~727px против ~465px доступных на 480px), поэтому дополнительно:

### **Разметка и навигация**
*   **Интеллектуальные линейки (`CanvasRulers`)** ([src/components/Editor/CanvasRulers/CanvasRulers.tsx](components/Editor/CanvasRulers/CanvasRulers.tsx)): Автоматически нумеруют ряды и колонны узловых точек с помощью `<text>` элементов SVG. Линейка по X (сверху) привязывается ко второму ряду узлов (r=1), чтобы корректно учитывать горизонтальное смещение нечётных рядов. Линейка по Y (слева) показывает номера рядов из первого (чётного) столбца.
    *   **Per-row Span Control (левое поле):** На левом краю между двумя соседними рядами узлов отрисовываются ± кнопки и счётчик для управления per-row span override'ом (см. раздел 2). Кнопки имеют цветовую кодировку (амбер для нижних пролётов, синий для верхних). При нажатии — мгновенный пересчёт сетки. Дополнительно: строка с ключом `-1` (верхняя цепочка) располагается выше первого ряда и использует общий механизм `rowSpanOverrides`/`onRowSpanChange`. Строка с ключом `-2` (нижняя цепочка) — ниже последнего ряда, отображается только при включённом `BottomEdgeDecor.enabled`, и, в отличие от остальных строк, её ± кнопки управляют `bottomEdgeDecor.span` напрямую (`onBottomEdgeSpanChange`), не затрагивая `rowSpanOverrides`.
    *   **Decor Control (правое поле):** На правом краю выводятся ± кнопки и счётчик для управления числом декор-рядов каждой полосы (см. раздел 3 → *Промежуточный декор*). Диапазон: 0 (полоса отсутствует) ... 10. При клике счётчик показывает текущее число или 0 для неактивной полосы.
    *   **Mirror Axis:** При включённом Mirror Mode отрисовывается вертикальная пунктирная линия по центру (`x = maxX / 2`, где `maxX` — максимальная координата чётного ряда).
    *   **Сворачиваемость Per-row Span Control (≤767.98px):** На узких экранах столбик ±/счётчиков по одному на каждый зазор между рядами занимает весь холст по вертикали, поэтому он свёрнут по умолчанию (`spanControlsExpanded` в `CanvasView.tsx`, `useState(false)`) и раскрывается «ручкой» — `.span-controls-toggle`, лежащей ПОВЕРХ карточки холста у её левого края (не снаружи, не рядом), с шевроном (›/‹), меняющим направление при раскрытии. Видимость контролов передаётся в `CanvasRulers` пропом и завязана на CSS-класс `.span-ctrl-layer--collapsed` (`CanvasRulers.css`) — эффект даёт только внутри `@media (max-width: 767.98px)`, на более широких экранах контролы всегда развёрнуты независимо от состояния тоггла (та же схема «состояние есть всегда, эффект — только в брейкпоинте», что и у мобильной раскладки хедера).
        При сворачивании освобождается и место под скрытые кнопки: левый отступ SVG-группы (`offsetX`) — числовой параметр геометрии, а не CSS-свойство, поэтому CSS-медиа-запросом его не подвинуть. Единственное в проекте место, где брейкпоинт проверяется в JS, а не только в CSS, — хук `useMediaQuery` ([src/hooks/useMediaQuery.ts](hooks/useMediaQuery.ts), обёртка над `window.matchMedia`). `effectiveOffsetX` в `CanvasView.tsx` — `BEAD_THEME.gridDefaults.offsetXCollapsed` (60px, хватает только под номера рядов) при `isNarrowViewport && !spanControlsExpanded`, иначе обычный `offsetX` (120px); участвует и в `computeCanvasDim` (сужает сам SVG-холст), и в `translate` корневой `<g>` — на десктопе, где контролы всегда видны независимо от `spanControlsExpanded`, JS-проверка ширины экрана не даёт им наложиться на сетку.
        Ручка `.span-controls-toggle` — `position: absolute` (`left: 8px`, `top: 50%` + `translateY(-50%)`) относительно `.canvas__svg-frame` — обёртки-«клона» размера `.canvas__svg` (`display: inline-block`, `position: relative`, без собственных width/height), которая, в отличие от самой карточки, не имеет `overflow: auto` — поэтому ручка лежит поверх карточки и не уезжает при горизонтальном/вертикальном скролле сетки бисерин. Две предыдущие попытки фикса — `position: fixed` относительно viewport, затем flex-элемент рядом с карточкой (с подгонкой `left`/`width` под `padding-left` холста) — сажали ручку СНАРУЖИ карточки, что не совпадало с задумкой «ручка на холсте», и в обоих случаях выглядело как «ручка оторвана».
        Ручка осмысленна только у левого края сетки (там же живёт панель, которую она раскрывает) — поэтому она скрыта (`opacity: 0`, `pointer-events: none` через класс `.span-controls-toggle--hidden`) всё время, пока `.canvas__svg` проскроллен вправо (`scrollLeft > 0`), и появляется обратно не по таймеру, а только когда пользователь докрутит холст обратно до левого края. Слушатель — на `canvasContainerRef` (сам `.canvas__svg`, скроллящийся элемент), состояние `isScrolledFromLeft` в `CanvasView.tsx`.
        Подписи линейки (`.canvas__axis-text`) на ≤767.98px тоже мельче (20px → 14px, CSS) и слегка смещены — `axisMarginX`/`axisMarginY` в `CanvasRulers.tsx` (30→36 и 40→46 соответственно) растут через тот же `useMediaQuery`, т.к. `x`/`y` текста — SVG-атрибуты, не стилизуемые через media-запрос напрямую.
        Отступы карточки холста на ≤767.98px тоже сжаты (`.canvas`: `padding-top` 24→14px, `padding-left` 36→24px; `.canvas__svg`: `padding` 20→14px, `border-radius` 18→14px, ширина `calc(100vw - 60px)` → `calc(100vw - 40px)`) — сетке бисерин остаётся больше места. Высота карточки (формула с запасом под `.stats` снизу) не тронута — это не padding карточки, а зазор под другой элемент.
*   **Центрирование:** ViewBox SVG считается автоматически по максимальным координатам элементов с учётом технологических отступов (`offsetX = 120px`, `offsetY = 50px`; переход из px в логических координатах происходит в генераторе).

## 4. Аналитика и статистика (Total Count)

`CanvasStats` — `position: fixed` поверх нижней части холста, поэтому карточка `.canvas__svg` резервирует под неё место в своей высоте. Раньше резерв был захардкожен в CSS (140px / 110px на разных брейкпоинтах) — при большом числе цветов `.stats__list` переносился на неограниченное число строк и реальная высота панели превышала константу, наезжая на нижние ряды бисера (особенно на мобильном, где ширина уже и перенос чаще). Хук [src/hooks/useStatsReserve.ts](hooks/useStatsReserve.ts) меряет через `ResizeObserver` фактическое расстояние от верхнего края `.stats` до низа viewport и прокидывает его в CSS-переменную `--stats-reserve` на `.editor__viewport`, которую формула высоты `.canvas__svg` использует вместо константы (с той же константой как fallback). Сам `.stats__list` теперь — CSS grid высотой ровно в 2 строки (`grid-template-rows: repeat(2, auto); grid-auto-flow: column`) с горизонтальным скроллом (`overflow-x: auto`) вместо переноса: бейджи цветов заполняют колонки по 2 шт., и при их накоплении высота панели больше не растёт (только `--stats-reserve` может слегка меняться между брейкпоинтами), а лишние колонки уезжают за пределы видимой области со скроллом. Ширина колонки зафиксирована через `grid-auto-columns: minmax(colWidth, max-content)`, а `max-width` списка рассчитан как `N * colWidth + (N-1) * gap` — так после ровно N видимых колонок гарантированно появляется скролл, а не произвольное количество в зависимости от контента. Число видимых колонок до скролла зависит от брейкпоинта (mobile-first, синхронизировано с паддингами/gap самих бейджей, чтобы реальная ширина бейджа не расходилась с шириной колонки на границе диапазона): 320–767px — 2, 768–1023px — 3, 1024px+ — 4. Второй источник того же наложения — единица `100vh` в самой формуле высоты: на мобильных браузерах она означает "большую" (layout) высоту viewport, как будто адресная строка уже скрыта, тогда как `.stats` (`position: fixed`) всегда считается от реально видимой области — при показанной адресной строке эти два числа расходятся, и `.canvas__svg` получается выше, чем реально видно. `100dvh` (динамическая высота, отслеживает показ/скрытие адресной строки) переопределяет `100vh` там, где поддерживается браузером, устраняя рассинхрон; используется и в `html, body, #root` ([index.css](index.css)). Третий источник наложения (тот же симптом, отдельная причина) — формула высоты вычитала только `--palette-height` и `--stats-reserve`, но не собственный `padding-top` обёртки `.canvas` (50px/24px/14px на разных брейкпоинтах), который физически лежит между шапкой и самой карточкой `.canvas__svg`. Карточка получалась ровно на величину этого padding-top длиннее доступного места и своим нижним краем наезжала на `.stats` независимо от точности `--stats-reserve`. Формула на всех трёх брейкпоинтах ([CanvasView.css](components/Editor/CanvasView/CanvasView.css)) дополнительно вычитает соответствующий padding-top.

Компонент `CanvasStats` играет роль калькулятора материалов в реальном времени:
*   **Total Bead Count:** Общее количество всех бисерин (NODE + SPAN) в текущем проекте **плюс** бисерины всех подвесок (`pendantPlacements`), у которых есть и валидный шаблон, и живая нода-якорь на нижнем ряду (та же проверка занятости, что и в `PendantLayer`). Любое изменение параметров сетки, покраски или размещения подвесок мгновенно обновляет это число (`CanvasView`, `useMemo` над `beads`/`pendantPlacements`).
*   **Color Specification:** Модуль статистики анализирует Design Map и выводит список всех задействованных цветов с указанием точного количества бисерин для каждого из них — включая бисерины подвесок (цвет из `placement.colorMap[index]` или дефолтный цвет типа бусины). Незакрашенные бисерины учитываются по своему цвету по умолчанию (см. раздел 5). Это готовая спецификация для закупки материала.

### **Подсветка и замена цвета**

*   **Подсветка:** клик по цветному кружку цвета в `CanvasStats` подсвечивает на холсте все бисерины этого эффективного цвета (`designMap[id] || defaultColorFor(type)`) — переиспользуется существующий проп `highlighted` у `BeadView`/`CrossWeaveBeadView` (см. раздел 3 → *Штамп*), а у бисерин подвесок — тот же визуальный класс `.bead__highlight`, отрисованный напрямую в `PendantLayer` (подвески не проходят через `BeadView`). Повторный клик по тому же кружку или `Escape` снимает подсветку. Состояние `highlightedColor` — локальное в `CanvasView`/`CrossWeaveCanvasView` (не персистентное), поэтому подсветка не переживает переключение техники или перезагрузку страницы.
*   **Замена:** у каждого цвета с валидным hex-значением (`#rrggbb`; бакет `transparent` незакрашенных бисерин кнопки не имеет — семантически это не «материал», а его отсутствие) в статистике есть маленькая кнопка-иконка (`Replace` из `lucide-react`). Клик по ней **не открывает отдельный пикер** — перекрашивает все бисерины этого цвета в уже выбранный активный цвет (`activeColor` из палитры/кастомного пикера/EyeDropper в хедере, тот же цвет, которым красит карандаш), одним кликом. Кнопка задизейблена, если цвет бакета уже совпадает с активным (нечего заменять). Замена затрагивает и Design Map, и все `colorMap` подвесок — одним шагом истории: `useDrawing.applyPatch` принимает трансформацию для дизайн-карты и (для silyanka) отдельную для подвесок, пишет один снимок в общий стек Undo/Redo. Чистая логика замены (без React) — [src/utils/colorSwap.ts](utils/colorSwap.ts) (`swapColorInMap`, `swapColorInPendants`), покрыта тестами. Если заменённый цвет был подсвечен — подсветка снимается (старого цвета больше не существует).

### **Экспорт схемы в PNG**

Кнопка «Download PNG» (в `CanvasView`, правый нижний угол; стиль консистентен с `.color-picker__confirm`) сохраняет текущую схему как растровое изображение. Весь текст в экспортируемом изображении — на английском. Логика — в [src/utils/exportScheme.ts](utils/exportScheme.ts):

*   **Источник:** живой `<svg>` холста клонируется — это гарантирует попадание в файл всех слоёв (бусины, подвески, нумерация рядов/столбцов, ось зеркала) ровно в том виде, как на экране.
*   **Очистка:** из клона удаляются интерактивные и служебные элементы — span-контролы целиком (кнопки `±` и числа пролётов), хитбоксы, подсветка наведения, drop-таргеты и кнопки удаления подвесок.
*   **Кадрирование:** экранный `viewBox` отбрасывается; реальные границы содержимого замеряются через `getBBox()` (клон временно цепляется за пределами экрана), либо берутся из `options.contentBounds`, если вызывающая сторона передала его явно (так делает CrossWeave — см. раздел 6 → *Экспорт PNG*). Без `contentBounds` это исключает срез линеек, которые рисуются в отрицательных координатах относительно группы трансформации. Вокруг добавляется поле `CONTENT_PADDING`. Через `options.extraStripSelector` можно удалить из клона дополнительные элементы перед экспортом (сверх `STRIP_SELECTORS`) — CrossWeave прячет так незакрашенные бисерины.
*   **Фон:** рисуется фоновый прямоугольник цвета поля холста (`#0f172a`).
*   **Стили:** оформление, заданное CSS-классами (обводка бусин, шрифт нумерации), вшивается в SVG элементом `<style>`, т.к. внешний CSS при сериализации теряется.
*   **Легенда материалов:** под сеткой дорисовывается SVG-блок с заголовком «Materials — total beads: N», списком цветов и количеством бисерин для каждого.
*   **Растеризация:** SVG → `<canvas>` с множителем качества ×2 (резкость под печать) → PNG-файл `silyanka-scheme.png`. Зум на экспорт не влияет — используется натуральный размер.

## 6. Техника «Крестик» (CrossWeave)

Независимый MVP-режим ([useCrossWeaveProject.ts](hooks/useCrossWeaveProject.ts), [CrossWeaveCanvasView.tsx](components/Editor/CanvasView/CrossWeaveCanvasView.tsx)): простая прямоугольная сетка овальных бисерин (`bead-r-c`), без node/span/decor/pendant-концепций силянки — карандаш, ластик, заливка и Mirror Mode. Этот раздел документирует Mirror Mode и заливку; остальная функциональность техники в спеке не описана.

### *Заливка (Flood Fill)*
Активируется кнопкой `PaintBucket` в хедере (между ластиком и Mirror Mode), шорткат `G` (тоггл, работает и в CrossWeave, и в силянке). Поведение UI идентично силяночному — крестик-курсор, один клик красит всю связную область того же эффективного цвета за один шаг истории (см. раздел 3 → *Заливка (Flood Fill)*).

**Алгоритм** ([src/utils/crossWeaveFloodFill.ts](utils/crossWeaveFloodFill.ts)): своя, более простая реализация, чем силяночный `floodFill.ts` — тут один тип бисерины и нет node/span/pendant, поэтому не нужен граф с цепочками, только прямая физическая смежность. Бисерина `bead-r-c` соединена с двумя бисеринами следующего ряда (`r+1`, другой ориентации): вертикальная (`r` чётный) — с горизонтальными `c` и `c+1`; горизонтальная (`r` нечётный) — с вертикальными `c-1` и `c` (та же чётность, что и в `crossWeaveGenerator.ts`/`crossWeaveMirror.ts`). BFS по этому графу, `effectiveColor = designMap[id] ?? defaultColorForCrossWeave()`.

**Mirror Mode:** если включён, заливка выполняется также для зеркальной бисерины (`mirrorCrossWeaveBeadId`); оба результата объединяются в один вызов `drawingControls.applyPatch` (второй аргумент — `null`, подвесок в CrossWeave нет).

### *Зеркальный режим (Mirror Mode)*
Включается кнопкой `FlipHorizontal` в хедере (сразу после ластика, аналог силяночной кнопки — см. раздел 3 → *Зеркальный режим*). Состояние персистентно в `localStorage` (ключ `crossWeave:mirrorMode`).

**Геометрия отражения** ([src/utils/crossWeaveMirror.ts](utils/crossWeaveMirror.ts)): у CrossWeave один тип бисерины, поэтому формула проще, чем `mirrorBeadId` у силянки. Ориентация ряда чередуется по чётности `r`, а не задана типом бисерины:
*   **Нечётный ряд** (горизонтальный, не смещён, `rawWidth` бисерин): `col c ↔ col (rawWidth-1-c)`.
*   **Чётный ряд** (вертикальный, смещён на `pitchX/2`, на одну бисерину короче — `rawWidth-1` бисерин): `col c ↔ col (rawWidth-2-c)`.

`rawWidth` — сырая ширина генератора (`logicalWidth + 1`, см. комментарий `toRawDimensions` в `useCrossWeaveProject.ts`), а не то число колонн, что показывает хедер/линейка.

**Покраска:** клик/мазок карандашом или ластиком закрашивает бисерину и её зеркальную пару через `applyPaint` в `CrossWeaveCanvasView.tsx` (аналог `applyPaint` силянки) — оба изменения проходят через один `paintBead`, поэтому попадают в один шаг истории Undo/Redo неявно (как и в обычном мазке).

**Ось симметрии:** при включённом режиме отрисовывается вертикальная линия (класс `canvas__mirror-axis`, переиспользован из силянки) по центру самого широкого (горизонтального, нечётного) ряда: `x = maxX / 2`.

**Изменение ширины в Mirror Mode:** как и у силянки, кнопки Width ± меняют ширину на **±2** (по колонке с каждой стороны), а прямой ввод числа округляет нечётную разницу до чётной — рисунок остаётся по центру. Design Map пересчитывается функцией `shiftCrossWeaveDesignMapColumns(designMap, delta, newRawWidth)` из того же файла: сдвигает `col` каждой бисерины на `delta`, бисерины за границей сетки отбрасываются. Пересчёт идёт через `remapDesignMap` и пишется в общую историю (Undo/Redo).

### *Экспорт PNG*
Переиспользует `exportSchemeToPng` из [exportScheme.ts](utils/exportScheme.ts) (см. раздел 4 → *Экспорт схемы в PNG*) через необязательный `options: ExportSchemeOptions`, но с рядом отличий от силянки:
*   **Легенда материалов скрыта:** `options.hideLegend = true` передаётся всегда — в отличие от силянки, блок «Materials — total beads: N» в CrossWeave-экспорт не попадает никогда.
*   **Незакрашенные бисерины скрыты, линейка вырезается всегда:** `options.extraStripSelector` — это `'.bead--empty, .canvas__ruler-group'`, когда закрашена хотя бы одна бисерина (класс `bead--empty` вешается в `CrossWeaveBeadView.tsx`, `isEmpty = !color`), либо просто `'.canvas__ruler-group'` на пустом холсте. Группа линейки удаляется из клона в обоих случаях.
*   **Кадрирование по явным границам** (`options.contentBounds`, считаются в `CrossWeaveCanvasView.tsx` как `paintedBounds`): `minX/minY/maxX/maxY` считаются одинаково — как минимум/максимум координат закрашенных бисерин (с учётом `beadMajorRadius`) — то есть обрезка симметрична по всем четырём сторонам bounding box'а закрашенного узора, а не только по правому/нижнему краю. Причина, по которой не используется `getBBox()` всего SVG: `CrossWeaveRulers` рисует подписи для *всех* рядов/колонок грида независимо от закраски, поэтому bbox всего клона всегда растягивался бы на весь холст — но поскольку сама группа линейки в любом случае вырезается перед растеризацией (см. выше), вопрос сохранения видимости её подписей не актуален.
